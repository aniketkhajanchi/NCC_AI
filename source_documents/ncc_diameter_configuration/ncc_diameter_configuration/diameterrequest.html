This action allows you to change the application id of the message to the value specified. The value of the new Application-Id must be zero or greater.

Changing the Diameter Application-Id and/or the command-Number information can potentially make the content of a message invalid to another device. Nokia recommends that you contact Technical Support if you need to use these actions.

Set-Proxy-Flag Sets the P flag. If set, the message may be proxied, relayed, or redirected. If cleared, the message must be locally processed.
Set-Retransmission-Flag Sets the T flag. This flag is set after a link failover procedure, to aid the removal of duplicate requests. It is set when resending requests not yet acknowledged, as an indication of a possible duplicate due to a link failure. This bit MUST be cleared when sending a request for the first time, otherwise the sender MUST set this flag. Diameter agents only need to be concerned about the number of requests they send based on a single received request; retransmissions by other entities need not be tracked. Diameter agents that receive a request with the T flag set, MUST keep the T flag set in the forwarded request. This flag must not be set if an error answer message (protocol error) has been received for the earlier message. It can be set only in cases where no answer has been received from the server for a request and the request is sent again. This flag must not be set in answer messages.

Suppress-Route-Record Used to suppress the route-record AVP. The value for this property is a comma-separated list of the origin-host (for example, Diameter identity) of the gateways that do not support the route-record AVP. When a request or response is sent to any of these gateways, the route-record AVP will not be included in the message. Suppressing the route-record AVP is particularly important when the Diameter client is a Cisco node.

Example of manipulating the generic processing of a DiameterRequest messages

For example, NCC default validation of destination realm on incoming requests allows an incorrect realm to be processed. It ignores the destination realm if the wrong destination realm is received. If you want a more strict validation with error treatment, you can author Diameter rules to apply a stricter validation that returns the error code and vendor ID. The following is an example of how to configure a rule to protect the system from messages being sent to an 'unknown' destination host and/or realm.

In the Message Validation GUI, configure a message validation with the following conditions and actions:

DiameterRequest.Destination-Realm

NOT_IN

(values)Realm1, Realm2, Realm3

Add Criterion

DiameterRequest.Destination-Host

NOT_IN

(values)Host1, Host2, Host3

Add Action

Discard

Add Action

DiameterRequest

Reject-Request

Result-Code = 4001

Vendor-ID = 0
