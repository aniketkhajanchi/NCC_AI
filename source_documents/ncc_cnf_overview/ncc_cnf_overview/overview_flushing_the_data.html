Flushing data

If necessary, the Redis database can be emptied in a controlled manner by executing the FLUSH_DATA instance of CG Configuration Node in the Configuration stream.

Note:
While the node provides two modes of flushing data: immediate flush and flushing of completed sessions, only immediate flush, which is the default option, should be used, because flushing of completed sessions is an obsolete option in a NCC system.

Immediate flush stops all incoming traffic, triggers session timeout for all the sessions currently in the database and processes the sessions normally, so the records end up in the correct CDR files. During immediate flush, Kafka input stream stops consuming incoming messages.

Flush mode is configured with CG Configuration Node parameter FlushMode, and flushing is done by executing the node.

When the flushing finishes and the database is empty, input streams do not start accepting incoming traffic automatically. Instead, CG Configuration Node must be run again, with node parameter FlushMode set to None, after which incoming traffic is accepted again.

For more detailed instructions on flushing, see the Operations, Administration and Maintenance Guide in the NCC documentation set.
