Rolling back an addition of independent pairs of 1+1 ME subsystems

This procedure provides instructions on how to roll back if an issue occurs while you are adding new independent pairs of 1+1 ME subsystems.

  1. Depending on the type of the issue that occurred when you were adding a pair of MEs, define the namespacess and NCC Telemetry instances that need to be rolled back.
    Note:
    You need to apply this procedure only to those namespaces and NCC Telemetry instances for which you completed the procedure Updating topology when adding an independent 1+1 ME pair. For example, if the topology update procedure was done for the NCC Telemetry instance, SMs, and MEs, but not for RTs, then you do a rollback only for the NCC Telemetry instance, SMs, and MEs.
  2. Define the order of the roll back, which depends on the system types and also whether you need to roll back NCC Telemetry instances.
    Note:
    If you have completed the topology update procedure for all components and systems, then do a rollback in this order:
    1. NCC Telemetry
    2. SM
    3. ME
  3. Note:
    If at any point during the rollback, an intervention is required, then first delete all the subscriber entities homed in the newly added ME.
    For each namespace and NCC Telemetry instance that you need to roll back, follow this procedure: Updating topology when rolling back an addition of independent 1+1 ME pairs in the correct order.
  4. For the last SM from in the rollback, clean up ME entry from SM aql database as follows:
    Note:
    If you do the rollback on two SMs, then this step is required only after the rollback of the second SM. This step is not required for NCC Telemetry, RTs and MEs.
    Execute the following on each newly added ME:
    kubectl exec -t <<admincli>> -n <<namespace>> -c <application_main_container> -- /opt/tpa/bin/db-record -d -n dsc -s SmManagedElement -k sps-me -f
    where sps-me is the name of a new ME.
  5. For each namespace that you rolled back, perform a health check:
    • Check the pods status using by entering:
      kubectl get pods -n <<namespace>>
    • When all the pods are UP and running, do the following to check the overall system health:
      • Check that the sps-status is UP, all vital services are running by entering:
        kubectl exec -it <<admincli>> -n <<namespace>> -c <application_main_container> -- /opt/tpa/bin/sps-status
      • Ensure that network-status does not show the added MEs now as they are removed by entering:
        kubectl exec -it <<admincli>> -n <<namespace>> -c <application_main_container> -- /opt/tpa/bin/network-status
      • Ensure that the partition status is returned correctly by entering:
        kubectl exec -it <<admincli>> -n <<namespace>> -c <application_main_container> -- /opt/tpa/bin/network-status -p
      • Ensure that the xdr-status is CURRENT by entering:
        kubectl exec -it <<admincli>> -n <<namespace>> -c <application_main_container> -- /opt/tpa/bin/xdr-status
      • Check that the db-stats is OK with all nodes responding and no migration in progress by entering:
        kubectl exec -it <<admincli>> -n <<namespace>> -c <application_main_container> -- /opt/tpa/bin/db-stats -c -s all
      • Check that there are no unexpected alarms on the log viewer.
      • Check that the new MEs are not visible on the Managed Elements section of the SM GUI anymore.
