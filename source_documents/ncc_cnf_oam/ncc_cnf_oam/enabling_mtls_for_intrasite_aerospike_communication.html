Enabling mutual TLS for intrasystem Aerospike communication

This procedure provides instructions on how to enable intrasystem mutual TLS (mTLS) communication on a particular system that occurs as server-to-server between Aerospike pods, and as client-to-server between Aerospike pods (servers) and the other pods (clients). This procedure is used post-installation when NCC has already been installed. For instructions on how to configure mTLS for intrasystem Aerospike communication during installation, see the Installation and Upgrade Guide in the NCC documentation set.

  1. Set these parameters in the values.yaml file:
    • aerospike.network.intraCluster.exposeInsure to true
    • aerospike.network.intraCluster.mtls.enabled to true
    • aerospike.network.intraCluster.mtls.credentialName to the name of the secret
    • aerospike.network.intraCluster.mtls.cipherSuites to DEFAULT
  2. Run helm upgrade with -f option passing the input values.yaml file to apply the changes.
  3. Restart the Aerospike server pods to upgrade the servers.
  4. Set these parameters in the values.yaml file:
    • global.aerospike.mtls.client.enabled to true
    • aerospike.network.intraCluster.exposeInsure to false
  5. Run helm upgrade to apply the changes.
  6. Restart the Aerospike server pods to upgrade the servers.
    Note:
    Aerospike communication uses a certificate for the aerospike-credential secret. When the current certificate expires or is rotated, the ongoing client-to-server connections are not affected. However, new connections require the updated certificate. To apply the updated certificate, the application and Aerospike pods must restart.
