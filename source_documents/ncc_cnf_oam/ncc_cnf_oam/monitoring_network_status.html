Monitoring Network Status

Heartbeat overview

This section describes the NCC network heartbeat, which enables NCC systems to construct a holistic and robust view of the status of the NCC systems in the network. The resulting view is more complete and accurate than if a given NCC site were to construct this view based on only its own visibility and assumptions. This network view is used to determine which sites are in-service and out-of-service relative to handling call processing traffic and in updating routing tables to ensure messages are routed to the currently active home for any given subscriber or device.

Location and community

Heartbeat uses location data configured in Topology to communicate status about connectivity and state. That is, Topology data is used in the determination of whether or not a system is in service. A site (location) is a building, complex, or buildings in close proximity that house one or more NCC systems (IG and/or MEs). Topology data is configured on each NCC system in the network to identify the location of all NCC systems with which the IG or ME exchange heartbeats. For more information, see the Installation and Upgrade guide.

In a multi-site network deployment (sites >= 3), heartbeat requires at least one other member outside its own site to determine if another member is OOS when it is missing a heartbeat with a member. For example, members in close proximity of each other (a site) may experience similar network connectivity issues. When connectivity to one member is lost, the lack of visibility of that member from another site must be confirmed before the member can be declared OOS.

Note:
Site information is used if there are three or more sites. Site information is not used if there is less than three sites.

In a two-site network deployment with two ME subsystems, if the heartbeat is missing between members, one system takes over as soon as the configured number of heartbeats is missed in one of the systems. However, in the case of a true split brain condition some manual intervention to isolate one system or the other may be required to set one of the members In-Service. For more information about isolating a system, see Forcing a system to be In-Service.

When does an NCC work on behalf of another NCC (reroute)

An ME subsystem always handles its own traffic, unless it goes OOS, which is a local decision. An ME subsystem takes over traffic for a remote system that it shares a partition with if the other system declares itself OOS, or if a community decision is made that it is OOS. NCC reroutes, or work on behalf of another NCC system when there is no OAM and Diameter heartbeat from an NCC system that is unreachable from any other NCC system. If one or the other heartbeat connection is not present, then heartbeat processing uses the status that is presented on the remaining heartbeat connection, which could present the system as In-service. At least one member from outside of the unreachable system’s own site determines whether or not the system is truly out-of-service (OOS). If the system is truly OOS as determined from the remote system, the NCC system is treated as OOS by the other members and traffic is rerouted. The SM and ME subsystems only use the secondary NCC when the primary NCC has zero indications of life from its own Diameter connectivity (ME subsystems only) and from the reports of the other NCC systems.

If ME 3 reports itself as OOS, all other NCC systems switch to the secondary ME subsystem for transactions normally homed on ME 3. If ME 3 reports itself as In-Service, transactions normally homed on ME 3 continue to be routed to ME 3.

All NCC systems provide their view that ME 3 is OOS. All NCC systems build a view that themselves and all others list ME 3 as OOS. All NCC systems switch to the secondary ME subsystem for transactions normally homed on ME 3.

Heartbeat in a geo-redundant NCC network deployment

Site (location) information in the NCC configuration files enhances the heartbeat component to use that location information for determination of status–IS or OOS. First, NCC determines if there is a third site from which to receive confirmation of a heartbeat, and second, NCC uses that confirmation from any NCC system at a third site to determine that a system not sending heartbeats is actually OOS.

Information that is communicated in the heartbeat is used by:

  • The Timed Event Manager (TEM) to determine whether the Primary system should process events for its Secondary system, when the Secondary is determined to be OOS. Session audits and monthly account resets are examples of TEM events.
  • The Diameter site routing plug-in to make decisions on how to route Diameter requests based on the states of the Primary and Secondary systems.
  • The SM subsystem to monitor the status of the NCC systems it manages and the state of the other SM subsystem.

NCC waits for the receipt of all reports from all members or the next polling interval before processing and making a decision on whether to change the membership status and/or the network view.

To determine a system's component status as IS or OOS, each member NCC system in the network deployment provides its state and view of the other NCC systems as the heartbeat monitor polls for status. NCC tracks these items for polled status:

  • member's component status (IN_SERVICE, OUT_OF_SERVICE, UNKNOWN).
  • number of missed heartbeats with member over OAM.
  • number of missed heartbeats with member over diameter if applicable.
  • timestamp of last heartbeat with member (Used for logging).
  • timestamp of last attempted heartbeat with member (Used for logging).

To be determined truly out of service where there are three or more sites, at least one member of the community that is located outside of the unreachable system's own site must see the offending system as OOS.

Heartbeat polling

When NCC starts up, heartbeat loads the topology for the members and then attempts to poll them. Polling goes in rounds; each member is polled in every round. During a heartbeat round, a polled member's heartbeat responds with its subsystem (for example, ME or SM) states for itself and other members subsystem states for the others. Any members that do not return a response go through an evaluation to determine what state should be used for them. The initial state for its members’ subsystems is UNKNOWN until they are polled.

The initial status of a member's subsystem state starts as UNKOWN, and remains that way until a response is received. Then it transitions to the state that is returned in the poll, either IN_SERVICE or OUT_OF_SERVICE.

When polling begins, the heartbeat monitor queries the remote web service, over the OAM network, that is served on the floating IP address of the active/standby centralmanagement pods. Then the web service queries the local Heartbeat Monitor to get the local states for its components, as well as its view of the network and returns it. This figure shows the heartbeat poll over the OAM network.

Figure: Heartbeat poll over the OAM network

The Diameter network polling occurs in parallel with the OAM network polling. The centralmanagement pod triggers the request, and then the application server sends the Diameter message to the remote system. The application server on the remote system queries its local centralmanagement monitoring service to retrieve the subsystem statuses and return them. In the case that the remote centralmanagement monitoring service is in the middle of an activity switch, the application container will only return its own system's subsystem statuses without the network view as it does not have access to the information.

Figure: Heartbeat poll over the Diameter network

The poll is sent out to the subsystems of the remote member. If a heartbeat is received, the reported status is used; otherwise, a check is performed to see if there has been enough missed heartbeats to make a decision. If the missed heartbeat threshold has been exceeded, there is a check to see if any system has seen that member as up. If yes, then the member is treated as in service. If not, then all members polled see the member is OOS. Then a check is performed to see if the polling system can see another site to ensure it is not the problem. If all systems see the member is down, and the polling system can see at least one other site, the member’s subsystem is declared down, an alarm is raised indicating that the subsystem is OOS and there is consensus from the community.

In a site with three or more systems, when considering a decision as to whether or not a subsystem in a site is OOS and to avoid making decisions on stale data, the decision is based on the input of other community members with whom heartbeat data is shared. If one or both are received then the heartbeat was successful. If a system has missed heartbeats with another member and has exceeded the missing heartbeat threshold (Missed threshold), it uses its community’s view of the system to determine if the subsystem is OOS. A heartbeat is missed if no result (Diameter or OAM) is received in a round.

Member response to heartbeat polling

When polled, a member not only responds with its state, but it also responds with its view of the community. When one member misses a heartbeat with another member (member X) of the community, it searches all the other member's responses (from that heartbeat round) for their view of member X to see if it can reach a decision about member X's state.

When a system is polled, it can report the other member’s subsystem as being:

  • Unknown

    The initial state for its members’ subsystems is Unknown until they are polled. A member cannot report its own status as unknown.

  • In-service (IS)

    If the polling system has polled the member and if it reported itself as IS.

  • Out-of-service (OOS).

    If the polling system has polled the member’s subsystem and it has reported itself OOS, or if the polling system polled the member’s subsystem and the member has exceeded the configured threshold for missed heartbeats.

Logging

Heartbeat status information from other members of the community is logged in the SPSServer.log. Most logging is at the debug level, but status changes for systems are logged at the info level. At start up, there is no previous poll with members; therefore, statuses of Unknown are expected. After start up and the first polling, statuses are reported. The status is maintained in memory and logged.

Activity switches on centralmanagement pods

When an activity switch occurs on centralmanagement pods, the primary heartbeat service is down until the floating IP Address is switched over and the heartbeat service is started on the new active centralmanagement pod. The duration of this event could be longer than the time it takes to declare a subsystem out of service (number of missed heartbeats * polling interval).

To prevent other NCC members from erroneously taking over processing with the activity switch, the Diameter heartbeat continues to respond to incoming requests, but it only responds with the subsystem statuses of the local member. It does not communicate the network view as that is only maintained on the OAM heartbeat service on the centralmanagement pods and the Diameter heartbeat service cannot retrieve that information from the centralmanagement pods during the activity switch.

Heartbeat configuration

Note:
Whenever possible, NCC systems should be deployed at three geographically separated sites so that subsystems at two sites can agree that subsystems at the third site are OOS.

Configuration for the heartbeat is done with the instantiation of the product but also consists of setting the following application preferences:

  • Heartbeat Interval (seconds):

    Used to determine the frequency of polling. By default the heartbeat interval is 2 seconds. The range is 1 to 15 seconds. 

  • Heartbeat Missed Beat Threshold:

    Used to determine a threshold for the number of missed heartbeats before a system is declared down. By default, the number of missed heartbeats before a system is declared down is 3. The range is 1 to 5. To declare an NCC system OOS due to missed heartbeats, an NCC system requires one other NCC in a site that is different than its own, in deployments with at least three sites, to communicate via its heartbeat that the system in question is OOS.

Note:
Application preferences are configured in the Global Configuration application. See the section “Geo-redundancy application preferences” in the Global Configuration Guide for more information.
Note:
In an IG system (containing both SM and ME subsystems), the forced in service tool can be used on the system affecting all subsystems, or it can be used to force individual subsystems in service if necessary. For more information on the tool, see Forcing a system to be In-Service.

On-demand heartbeat status

Tools are available to get the heartbeat status on demand. For the local system status, use the network-status tool. For more information on the tool, see network-status.

Forcing a system to be In-Service

In a network deployment that consists only of a pair of IG systems, there may be a time when an NCC system cannot heartbeat with the other system and you must manually override the OOS decision that the heartbeat has made. This would be a situation where network connectivity is problematic because both the OAM and Diameter networks are severed between two sites, but both ME subsystems and SM subsystems are processing on behalf of the ones in the other system.

In a network deployment with two IG systems and two standalone ME systems, in a total of two sites, it is possible to enter a true split-brain condition. In this case, members from one site cannot see members from the other site, which causes them to incorrectly start processing on the other members behalf while they are still processing their own traffic and TEM events. When the communication cannot be restored, the force-remote-site-in-service tool can be used to treat the other member as In-Service so that processing on behalf of that member will stop. The tool overrides the heartbeat decision for the member that this tool is run on, until a direct heartbeat is received from the errant member.

In this example, when an NCC system in Site 1 cannot see an NCC system in Site 2, and the other way around, the tool needs to be run on all systems. When this tool is used to force a member IS, the "NCC treating other member as if heartbeat received In Service" alarm is generated. When the tool clears the heartbeat status, it raises "NCC heartbeat treatment returned to normal" and then clears the first alarm. 

The forced In-Service treatment ends when a heartbeat is actually received from the errant NCC, which indicates a status other than In-Service or after 10 minutes of In-Service heartbeats are received without missing the number of heartbeats that are required to consider it OOS. When this occurs, the "NCC heartbeat treatment automatically reset to normal" alarm is raised. If an activity switch of the centralmanagement pod occurs in the 10 minute window, the count is reset when the standby takes over.

The tool also provides an option to list any active forced In-Service members and an option to manually clear the forced In-Service state of a specific NCC. For more information on the tool, see force-remote-site-in-service.

Heartbeat alarming

Note:
Alarms are raised per monitored subsystem, so in an integrated system an alarm is raised against the SM and the ME subsystems individually. The "NCC unreachable Over Diameter" is an exception, as it does not differ based on subsystem.

There are different alarms that can be raised by the heartbeat monitor. The alarms cover scenarios where OOS is declared in a network with at least three sites, where a remote member is OOS, and where Diameter heartbeats are missed with a member. This is a list of how the alarms are named:

  • “NCC unreachable over Diameter” is the alarm raised when Diameter heartbeats are missed.
  • “NCC was determined to be Out-Of-Service” is the alarm raised when the heartbeat sees the remote member as OOS.

  • “Heartbeat is missing with member” is the alarm raised when there are more than three sites and a member is:

    • missing a heartbeat, but someone continues to see the member as IS.
    • missing a heartbeat, and everyone sees OOS, but no other location is seen.
    • missing a heartbeat, and everyone sees OOS, but another location is seen.

      Only one of above can be raised at a time.

If the deployment has less than three sites, the missing heartbeat alarms are raised when a member is:

  • missing a heartbeat, but someone continues to see the member as IS.
  • missing a heartbeat, and everyone sees OOS.

These scenarios describe the use cases where the above alarms are generated by the heartbeat monitor in networks that have at least three sites and where heartbeats with the other member have been missed.

  • Alarm 1: If the heartbeat between this system and the remote system is down, all members reporting a heartbeat agree that the remote system is unreachable or OOS. At least one of the reporters is from another site. The remote system is treated as OOS.
  • Alarm 2: If the heartbeat between this system and the remote system is down and all members reporting a heartbeat agree that the remote system is unreachable or OOS, but you can only see members at your site, the problem may be with the systems at this site. The member is treated by its last known status, which could be IS or OOS.
  • Alarm 3: If the Heartbeat is missing with member, at least one other member that received heartbeats from can see the remote system as in service.

In configurations with less than three sites where the heartbeat between this system and the remote system is down, the remote system is treated as OOS.

Only one of the above alarms exists per member with polling. If there is a status change to a more or less severe scenario, then the existing alarm is cleared and a new alarm is raised. A heartbeat alarm is cleared when the member no longer misses a heartbeat, or when the conditions of another alarm is met. If another alarm condition is met, the new alarm is raised in its place. If the heartbeat returns, any active alarm is cleared.

This example demonstrates how the alarms are raised and cleared when there are at least three sites in the network:

  • Multiple alarms can be raised but only one alarm per member. For example, in a four-member network, Member 1 does not have a heartbeat with Member 2 or Member 3, but is polling with Member 4. Member 4 sees both Member 2 and Member 3 as in service. On Member 1, these alarms are raised:

    • ALARM3 for Member 2
    • ALARM3 for Member 3

  • Transitions from an existing alarm to another alarm where Member 1 does not have a heartbeat with Member 2.  However, Members 3 and 4 continue to see Member 2 as in service.  The current alarm state is:

    • ALARM3 for Member 2

  • On the next heart beat round, Members 3 and 4 report Member 2 as OOS. Member 1 then clears the alarm and raises a new alarm.

    • Clear ALARM3 for Member 2
    • Raise ALARM1 for Member 2

  • An alarm is cleared where Member 1 has an alarm that member 2 is OOS and it is missing heartbeats (ALARM1). On the next heartbeat round, it receives a response from Member 2. Member 2 reports itself as OOS, but because there are no missed heartbeats with ME2, the alarm is cleared.

    • Clear ALARM1 for Member 2
  • Other alarms can also be raised:
    • NCC heartbeat treatment automatically reset: NCC automatically reset to normal heartbeat treatment for another NCC.
    • NCC heartbeat treatment returned to normal: NCC returned to normal heartbeat treatment for another NCC.
    • NCC treating other site as In Service heartbeat received: NCC forced to treat another NCC as if its heartbeat was received indicating In Service.
