Error CDRs for failed Gy and NCHF (N40) records

In the processing of Gy and NCHF records, the result codes of the received records can be checked so that the failed records can be directed into a separate output directory from successfully processed CDRs. A record is considered successful if the ResultCode tag on both root level and MSCC level contains successful codes.

Failed records are identified if either root level or MSCC level contains failed result codes. If the resultCode is missing on the root level, the record is considered a failed record. However, if it is missing on the MSCC level, the record is considered partially failed.

Records are separated into two output directories, one for failed records and one for successful, for Gy and N40 records respectively. Successful records follow the default behavior defined by distribution rules. See CDR mapping/distribution rule for more details.

Enabling error CDR generation

To generate error CDRs into a separate output directory, the following actions are needed:

  • Configure successful result codes in CG Mapper Node (Business logic stream).

    • For Gy sessions, configure the success result codes in CG Mapper Node parameter ResultCode. If needed you can define multiple codes (Integer and String values) by using a comma-separated list.
    • For N40 sessions, configure the success result codes in CG Mapper Node parameter GySuccessCode. If needed you can define multiple codes by using a comma-separated list.
  • Activate the feature.

    To take this feature into use, set the following node parameters in CG Mapper Node (Business logic stream):

    • EnableMsccSplitting: Set the value to Yes.
    • AggregateEnabled: Set the value to No.
