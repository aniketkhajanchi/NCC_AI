Subscriber healing and orphan lookup tables audit

The -k option in the db-list tool is used to provide an input file for finding inconsistency (-i option) and healing (-a option) for subscribers of a specific club.

For each line of the input file, club IDs are fetched based on the input parameters, then a consistency check and heal (if enabled) is performed base on the club ID result.

Notice:
Consult Nokia support before you use the -a option.

If you use the -a report option when running this tool, the tool only performs a consistency check.

If you use the -a heal option when running this tool, then based on the input parameters, the tool performs these actions:

  • If only one club ID is found in the InfrequentlyChangingData set and there is one record present with this club ID as the primary key, then a consistency check and healing (if enabled) are processed for this InfrequentlyChangingData record.
  • If only one club ID is found and InfrequentlyChangingData does not exist for this club ID or InfrequentlyChangingData for this club ID exists but no entities (device or account or group or user) exist in this InfrequentlyChangingData, then orphan lookup tables that are related to parameters in the input file are deleted.
  • If two or more club IDs are found and InfrequentlyChangingData does not exist for all club IDs or InfrequentlyChangingData for these club IDs exists and no entities (device or account or group or user) are present in the InfrequentlyChangingData set:
    • If you used the -s primary option when running the tool, if all orphan lookup tables belong to the local system, then orphan lookup tables related to parameters in input file are deleted. If one or more club IDs belong to a remote system, then only a consistency report is generated and no more actions are taken.
    • If you did not use the -s option or used the -s all option when running the tool, then all orphan lookup tables related to input parameters are deleted, no matter if they are local or remote.

Configuration file

The configuration file is located in the /opt/tpa/bin/db-list.properties directory.

This properties file contains lookup table names to be healed for a corresponding subscriber. Tables that are not included in this file, are not be healed.

These three parameters are included in the file:
  • ATTRIBUTE_DELIMITER - used to separate different types of IDs (the default value is a comma). It is supported to find inconsistent (-i option) and heal (-a option) data according to parameters included in the CSV file (-k option).
  • ATTRIBUTE_VALUE_DELIMITER - used to separate different values of an ID (the default is two colons). You should make sure these delimiters are not used in names or IDs and should not include the equals sign. This is an example with default values: 
    Clubid,group,group-id,group-name,parent-group-id,administrator-id-1::administrator-id-2
  • ACCOUNT_BALANCE_VALUE - When healing inconsistent data for an account, if InfrequentlyChangingData is present but DD is not present or the account balance is missing from its DD, the tool creates an account balance in DD in the same club of the account's InfrequentlyChangingData and sets the balance to the value set by the ACCOUNT_BALANCE_VALUE parameter. The default value is 0. Only positive values are supported. An error is thrown if a negative value is provided. If this parameter is not present in the db-list.properties file, no healing is done.

input file format

An entry for each subscriber-entity should be given in the input file. For subscriber-entities, the first four entries are common. The remaining entries are subscriber-entity specific. For example: 
ClubId,subscriber-type,Subscriber-id, subscriber-name,..,..,
  • Club id: should be numeric
  • Subscriber-type: device/group/account/user
  • Subscriber-id: ID of the subscriber
  • subscriber-name: name of the subscriber-entity
For a Device entity, the input file entry should be in this format: 
Clubid,device,device-id,device-name,e164-1::e164-2,imsi-1::imsi-2,msisdn-1::msisdn-2,nai-1::nai-2,sipuri-1::sipuri-2,
privateid-1::privateid-2,externalid-1::externalid-2,group-id-1::group-id-2,user-Id
For a Group entity, the input file entry should be in this format: 
Clubid,group,group-id,group-name,parent-group-id,administrator-id-1::administrator-id-2
For an Account entity, the input file entry should be in this format: 
Clubid,account,account-id,account-name,user-id-1::user-id-2
For a User entity, the input file entry should be in this format: 
Clubid,user,user-id,user-name,email-id-1::email-id-2,phone-number-1::phone-number-2,facebook-id
Note:
  • If the -i option argument does not match with the subscriber-type of the input file, validation failure information is included in the CSV output and no more action is processed. For example: if ./db-list -i devices -k inputfile is run and 1234,group,Group1,Groupname is present in the input file, validation fails and "subscriber-type inconsistent" is shown as the failed reason in the CSV output.
  • When the Application Preference AutoGen SPR Ids value is set to true, the Subscriber-id value in the input file should be in UUID format, otherwise orphan lookup tables related to the subscriber-id are not cleared.

Report formats

A CSV file is generated for the summary report. XML files are generated for the inconsistency and healing reports. This information is provided, based on entity type:
  • Entity-status: INCONSISTENT/CONSISTENT/ERROR/VALIDATION_FAILED::<reason for validation failed>/UNABLE_TO_PROCESS:<reason for unable to process>
  • Healing-status: HEALED/NOT_HEALED/ERROR/NOT_APPLICABLE/ORPHAN_DELETED
  • account-balance-status values: BALANCE_UPDATED/NOT_APPLICABLE
