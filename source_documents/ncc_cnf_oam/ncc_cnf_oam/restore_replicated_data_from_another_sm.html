Restore replicated Aerospike data

Disable Aerospike data replication from healthy systems

Some operations must be performed on the other healthy SMs in the deployment. This ensures the recovery process is not disrupted.

To obtain a list of the names of SM systems in the deployments, run this command from the admincli pod of any SM in the deployment:

/opt/tpa/bin/disaster-recovery-helper.py -w        
        ## This is an example of the output:
        INFO: This tool must be run on following list of sites : ['SM1','SM2','SM3','SM4']

On each healthy SM in the deployment, temporary update the replication configuration to exclude SM4 by running this command from the admincli pod:

/opt/tpa/bin/disaster-recovery-helper.py -a remove -s SM4 
## where SM4 is the name of the recovering SM system

Restore replicated data from mate system

The replicated data can be restored from any healthy system.

## Restart the xdrproxy pods on SM4, so the replicated namespace can be restored.
## Run this command, where $XDRPROXY_DEPLOYMENT is the name of xdrproxy deployment in your installation
kubectl -n $NAMESPACE scale --replicas=1 deploy $XDRPROXY_DEPLOYMENT
 
## On the healthy SM that you want to use to restore data on the recovering system (for example, SM1), perform a rewind of SM4.
## Run this command from admincli pod of SM1, where the value for -s option is the name of the recovering system SM4.
/opt/tpa/bin/disaster-recovery-helper.py -x start -s SM4
 
## On SM1, check the status of the rewind. 
## Run this command from admincli pod of SM1, where the value for -s option is the name of the recovering system SM4,
## and the value of -l is the max number of times the script should check for rewind status, at intervals, before exiting. Default is 100.
## The script can be run multiple times until it reports that the rewind is complete.
/opt/tpa/bin/disaster-recovery-helper.py -x status -s SM4 -l 20
 
## Once the rewind is complete, go to the recovering system SM4 and make sure the Aerospike pods are in running state.

Re-enable Aerospike data replication

## On each SM except the recovering system SM4, restore replication to SM4. Run this command on all healthy SMs, from the admincli pod:  
/opt/tpa/bin/disaster-recovery-helper.py -a add -s SM4

Clear DOWN_DRR state for Aerospike cluster

# NOTE: This clears the DOWN_DRR state in case if it was set for the system. 
kubectl -it exec <admincli> -c <container> -n <namespace> -- /opt/tpa/sbin/db-roster.py clear

Repair the key generators data set

Run this command on the recovering system (only for IG).

kubectl -it exec <admincli> -c <container> -n <namespace> -- /opt/tpa/bin/TPA_repairKeyGen
