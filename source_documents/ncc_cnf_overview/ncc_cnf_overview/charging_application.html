Charging application

This section provides the overview of the Charging application. For more information about the Charging application, see the Charging User Guide in the NCC documentation set.

The Charging application receives and processes various types of network requests related to subscribers and performs rating and charging functions inside NCC. The application performs these functions:

  • subscriber rating and charging based on the different criteria and rules (using A.R.T rules) that are configured by operators as per their market offers
  • supports group solutions ranging from a small group to a big group
  • supports MVNO for multi-tenant solutions
  • enables service operators to control their offers and interactions with subscribers
  • addresses the data market based on advanced 5G HTTP/2 and native 4G Diameter interface
  • supports IMS/SMS/MMS services over 4G Diameter protocol
  • performs the role of an Online Charging Function (OCF) or Charging Function (CHF) for 4G and 5G interfaces respectively
  • supports 5G interfaces N40, N28, N41 and N44 by interacting with SMF and PCF. Interacts for NRF registration for 5G interface.
  • performs service and subscriber-related configuration over REST interface
  • supports IoT devices use cases over 5G. For more information, see the Charging User Guide and Standard Compliance and Interface Reference.
  • supports a set of TMF standards API. For more information about TMF API, see TMF Open API Specifications in the NCC documentation set.
  • capability to act as an NCC router for 5G data traffic where 5G requests (N40) are converted to 4G diameter (Gy) and then forwarded to legacy 4G OCS of the operators based on certain configurable rules. It also generates CDR based on the 32.298 format for such requests. For more details about 32.298, see the CDR/EDR Reference.

The Charging application uses Ro and Gy Diameter interfaces as native interfaces for rating and charging for various types of subscriber requests like data, voice, SMS & MMS services. The charging application supports the advance 5G interface for rating and charging of data services. These subscriber events can either be session-based or event-based. The application uses Sy (4G) and N28 (5G) interfaces to interact with PCRF/PCF to enforce subscriber policy decisions.

For more information/compliancy about 4G/5G interfaces, see the Charging User Guide and Standard Compliance and Interface Reference.

Charging application components

The Charging application is composed of several modules that interact with each other. This table describes the components.
Table: Charging application components
Component Description
Call controller This component encodes and decodes the 4G Diameter and 5G HTTP/2 request/response from/to the network. This component interfaces with the rating engine to perform the rating and charging of subscriber requests based on the different criteria and rules configured by operators as per their offers.
Subscriber model This component is responsible for managing the subscriber and group-related information inside the NCC database.
Service model This component is responsible for managing the charging and rating related data objects in the NCC database such as bundles/services, tariffs, buckets etc. The operator configures these various data object to create offer/product catalogue based on their market needs.
Session management This component is responsible for creating, updating, deleting, and auditing the call sessions from memory.
Life cycle NCC has a flexible lifecycle model where a variety of lifecycles can be created for different offers for both prepaid and postpaid solutions. This component manages and controls the renewals and validity of products that are purchased by subscribers based on the lifecycle that is associated with those products.
Charging and rating engine

This component is responsible for performing a correct rating and charging of a subscriber event/request based on the different criteria, rules, rates, and tariffs that are configured by the operators.

It generates CDR/EDR records as part of subscriber request processing which is then sent to the CGF (DR) application via the Kafka channel. The CGF (DR) application writes these CDR/EDR records in the file which are then used by external billing mediation for subscriber billing purposes. CDRs are generated based on the generic ASN that is defined in the CDR/EDR Reference.

As part of the subscriber event/request processing, it generates the notification events, if applicable, towards the Notification application. The Notification application sends the notification to subscribers via different channels like SMS/EMAIL/REST/Kafka etc.
