Alarm administration

Overview

Alarms are generated to alert operators of abnormal behavior. Alarms are raised by any of the NCC microservices. These alarms are meant to be sent to the Element Management System (EMS) or Operations Support System (OSS) that displays them.

The alarming service is provided across a cluster of systems in a cloud environment and supports:
  • southbound interface that allows applications to generate alarms.
  • alarm management that includes alarm processing, storage and lifecycle management.
  • northbound interface (SNMP v2c/v3) that reports alarms to the EMS or OSS.

Alarm Service GUI

Alarms are stored in MariaDB and can be viewed in the Alarm Management GUI. To access the Alarm Management GUI, navigate to https://$FQDN/calm-<site type>.

Example:

https://$FQDN/calm-sm

Users can log in to the Alarm Management GUI with their configured username and password. The GUI displays a list of all active alarms. There is an option to see all of the cleared/historical alarms. Alarms can be exported in CSV format from the Alarm Management GUI. You can also perform other basic operations on the alarm list such as sorting, filtering, managing table columns.

Note:

For alarm administration, an enterprise label is present for each alarm. The value that is present in this label depends on the service provider (whether the alarms support MVNO or not and if yes, whether MVNO is enabled by the application or not).

To obtain the details about MNO, MVNOs, and how to enable them; see the Service Provider section of the Global Configuration Guide.

There are three conditions based on which the value of enterprise field is defined:

  1. If MVNO is disabled, the enterprise field is empty for all of the alarms regardless of whether they support MVNO or not.
  2. If MVNO is enabled and the alarms support it, the value of the enterprise field is the MVNO for which the alarm was raised.
  3. If MVNO is enabled but the alarms do not support MVNO, the value of the enterprise field is blank.

The alarms with an Enterprise label value of yes support MVNO.

Active alarm table and alarm logs

The size of the active alarm table is configured with the maxActiveAlarmCount key. By default, the size of the active alarm table is 4096 active alarms.

If the storage backend is down, the system sends a trap to the SNMP manager with alarm id 2111111111 trying to reach MariaDB. If the system is not able to communicate with MariaDB, the system stops processing alarms or sending traps of any alarm.

The file /var/log/calm/alarms.log is used to log the dropped alarms that exceed the alarm table size set in the configuration file alma.properties.

Note:
Additional text will be missing for fs_mon alarms and severity is the only indication you see.

Brevity control

Alarms of a similar nature may be triggered at the same time. To a northbound management system, there can be a burst of alarms arriving which can cause overload. For example, there are multiple instances of a service running in a cluster when a network partition occurs. If half of your service instances can no longer reach the database, each service instance sends an alarm if it cannot communicate with the database. This condition results in hundreds of alarms being sent to the northbound management system. Brevity control allows you to configure and send a limited number of alarms that are of a similar nature but are enough to help identify the root cause of what is happening at that time and avoid going into overload.

Brevity control applies to alarms and only allows a specific number of the same alarms to pass, dropping the rest of the alarms that are created within the particular interval.

Using brevity control

You can enable and define the brevity rule in the input generator (Inputs.xlsx file in the BTEL section of the Common tab). The Alarm Service maintains statistics on incoming raise alarm events that match this specific rule's target. These statistics are used to determine whether or not alarms are blocked. Incoming alarms are counted for distinct thresholdPeriod length time intervals.

A brevity rule defines the parameter values (upperThreshold, lowerThreshold, thresholdPeriod) which are applied for the specified targeted alarms. You cannot specify different rules for different alarms. These fields must be filled out if the CALM Brevity Control Enabled value is set to true.

  • target (alarm_id, moc_id): comma-separated list of alarm IDs that are targeted. The rule is applied to all incoming matching alarms. The target value consists of an alarm_id (source problem) or alarm_id:moc_id.
  • upperThreshold: upper threshold for the number of incoming raise alarm events within a threshold period. It is used to determine if alarm blocking should be started.
  • lowerThreshold: lower threshold for the number of incoming raise alarm events within a threshold period. It is used to determine when to stop blocking alarms.
  • thresholdPeriod: length of the threshold period (in seconds). The number of incoming raise alarm events are counted within threshold periods and checked against the upper and lower thresholds.

Brevity control starts to block alarms when the number of incoming alarms exceeds the upperThreshold in one single threshold period. For any subsequent activation of alarm blocking, the number of incoming alarms has to exceed the upperThreshold in five consecutive periods. During blocking, the incoming raise alarm events are dropped. Alarm Manager creates a specific alarm that indicates the start of blocking alarms. Until the blocking is canceled, the brevity control alarm status is updated with the current statistic after every five periods.

Brevity control cancels blocking alarms if there was an incoming clear alarm event that matches the rule or the number of incoming raise alarm events for a threshold period drops below the lowerThreshold. Alarm Manager clears the brevity control status alarm (that was created when the blocking of incoming alarms started).

Brevity rule keys are used to match incoming alarms with the defined brevity rules. If there is a matching rule, it is applied. When the rule is created, the contents of the target field and the name of the enterprise are used. For an incoming alarm, the rule key is constructed using the corresponding fields of the alarm. (Here the alarm id is the specific problem).

When multicontext is enabled, the brevity rule key is constructed as enterprise:alarm_id:moc_id.

The enterprise part is omitted when multicontext is disabled: :alarm_id:moc_id.

The Brevity control message for dropped alarms is displayed when the second brevity control occurs on the same alarms.

Alarm forwarding to a webhook receiver

The Alarm Service can be configured to forward alarms to a webhook receiver in JSON format. Each namespace worksheet in the Inputs.xlsx file has an Alarm Webhook section where you can specify the webhook targets (maximum of two), number of retries, and retry delay. You cannot mix secure with unsecured webhook targets. You can forward alarms to both an SNMP receiver and webhook receiver at the same time.

Alarm forwarding is done in parallel if two webhooks targets are configured. There is no concept of a primary and secondary webhook target.

The Alarm Service tries to forward any alarms within the retry timeout to the webhook receiver if the target is down or unreachable.

The Alarm Service maintains a queue for all notifications that are sent out successfully. If this queue fills up, then further messages are not sent. As soon as the queue size decreases below the maximum, notifications are sent out again. The default queue size is 1000 for lab/dev environments and 3000 for production environments. By default, the service uses three threads to send out notifications. In cases of high load, the maximum number of threads is 10.

You must create an Alarm Service override file to change the values for the queue size and maximum number of threads values. Edit the calm.webhook.queue-capacity and calm.webhook.max-number-of-threads values in the override file and perform an upgrade.

SNMP alarms

NCC supports a framework for producing SNMP alarms. When an alarm is produced, NCC maintains an active state until the alarm condition is cleared.
Note:
NCC generates an alarm timestamp in SNMP traps that can be up to five seconds after the actual event time.

For more information regarding SNMP alarms, see SNMP configuration.

Alarm trap

The alarm service provides the ability to send alarms as traps to the SNMP manager.
Note:
To specify the SNMP manager, you need to configure the SNMP users parameters after instantiation. To configure the SNMP users, contact Nokia Professional Services.

Alarm identification, uniqueness, field descriptions, and parameters

Alarm uniqueness is a composite of the following parameters:
  • System
  • SpecificProblem

In the NE3S protocol, "noiSystemDistinguishedName" + "noiSpecificProblem" can only define one alarm. So in the API, "system"/"host"/<"container">/<"service">/<"process">+"alarm.id" defines one alarm only. The "system" and "host" are mandatory. The others are the optional. ObjectName is constructed using <host>+<service>#<system>.
