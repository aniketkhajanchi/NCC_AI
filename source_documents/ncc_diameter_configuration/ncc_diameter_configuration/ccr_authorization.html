CCR authorization in the NCC Gy interface

No MSCC in CCR-I is considered as CCR authorization. If device is eligible (not in barred state) for call, then 2001 is sent in CCA. Otherwise, an appropriate error result code is sent and session is not created.

Application preference

Description

Data type

Default value
Gy Session Authorization Validity (seconds)

If it is set to 0, then CCR Authorization feature is considered as disabled; otherwise CCR Authorization feature is enabled.

Its value specifies the time in seconds after which the Gy session created due to CCR Authorization is closed if no further request is received for that session.

 integer 0
Clear session if no RG active

Currently, the result code at command level in CCA is always set to 2001 irrespective of the result code values at MSCC level. Therefore, a new configurable boolean parameter named clear session if no RG active has been introduced in charging and its applicable only for CCA-I.

If this flag is false, then the current behavior where CCA is always 2001 is followed.

 Boolen false

If this flag is true and an active MSCC or RG does not exist for a given session, then perform the following task:

  • Set the MSCC level result code at command level, if only one MSCC is returned in CCA-I.

  • Set the same error result code at command level and session is not created, if there are multiple MSCCs returned in the CCA-I with the same Error Result code.

  • Set the result code to 2001 at command level and session is created, if there are multiple MSCCs returned in CCA-I with different Error Result code.

    

There is no change in behavior for following cases (where CCA = 2001) irrespective of the value set for Clear session if no RG active parameter:

  • Event call behavior

  • CCR-T behavior where the root level result code is set to 2001 irrespective of the MSCC result code and always terminate the session

    
New Call Start in Update or Terminate During migration from existing charging system to NCC, CCR-U or CCR-T for ongoing session on existing charging system arrives on NCC. As Gy sessions for these CCR-U/T do not exist on NCC, these CCRU/T are rejected. To handle this case, the New Call Start in Update or Terminate attribute is introduced. Boolean false

If this flag is set to true, then NCC performs the following tasks:

  1. Accepts the CCR-U/T which are active on the existing charging system and treats them as CCR-I/T.

  2. Commits the used quantities which are reported in such CCR-U/T.

Valid values are true and false and the default value is false.

Note: Subscription-ID AVP is a mandatory parameter to be sent in CCRU/T request from the existing charging system to NCC. Otherwise, NCC rejects the call.

    

This authorization feature is controlled by configuration as mentioned in Charging application preferences section in the Managed Element application preferences chapter of the Global Configuration Guide.
