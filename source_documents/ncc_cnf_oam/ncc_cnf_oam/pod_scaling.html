Pod scaling

Perform these steps to scale out/in the pods in an NCC deployment.

Note:
  • Aerospike scaling causes traffic outage and should only be done with a geo-deployment setup.
  • Scale out/in activities should be planned by considering the engineering load on the site. Traffic is impacted if you scale in beyond the required dimensioning.
  • Perform scale operations during a maintenance window (low peak hours).
  • You must allocate resources before scale out activities.
  • For IOHD, Central Management is the site going Out-of-Service (OOS).

These are the basic steps in the process:

  • Health check
  • Increase/decrease the pod count by using Helm commands
  • Verification steps
  • Post-verification

Health check

Note:
If any health check fails, you must fix the issue and then proceed with scale operations.

All pod containers must be up and running before you start scaling operations.

Run these tools on the namespace to verify all pods are up and running:
  • sps-status - verify all components are UP.
  • network-status - verify all components are IN_SERVICE.
  • db-stats
  • xdr-status - verify both local site towards remote sites and remote sites toward local site

    have sync up state of CURRENT.

Note:
  • For HPA pods, you must increase the minReplicas: and maxReplicas: values to the desired count in the generated override files (for example: charts/chf-charging.yaml). (Horizontal Scaling)
  • For non-HPA pods, you must increase the replicas: value in the generated override files (for example: charts/chf-ckaf-kafka.yaml). (Horizontal Scaling)
  • For Vertical Scaling only: cpu, ephemeral-storage, and memory can be increased.
  • The pods restart during the Helm operation.
  • To scale MariaDB pods, increase the count variable in the generated override file. 
    mariadb:
  count: 1
  1. Update the replica count to the desired count of pods in the override files for the respective chart.
  2. Run this command to perform a Helm upgrade with new replica counts chart files. 
    helm3 upgrade --install -n <namespace>  <chart> -f <Updated_yaml_file_with_Path> --timeout 1800s
  3. Wait for the new pods to come up.
  4. Verify local cluster migration using admincli pod.
  5. Delete the PVC in case of scale in (as applicable).
  6. Perform another health check.

Rollback

After the scale operation, if any of the health check started failing or there is a need to rollback to the previous state, then perform the reverse operation.

For instance, if the charging pod’s horizontal scale out operation failed due to a resource issue and there is no possibility to provide extra resources, then re-run the Helm upgrade operation with the original override files for the charging pods.
