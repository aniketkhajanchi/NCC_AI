db-list

Description

The db-list tool lists the records in the Aerospike database and the listing is written into compressed (.gz) files in a JSON format. The tool can also be used to audit the data to report inconsistencies and optionally to heal the data when feasible.

These guidelines are applied for the usage of the tool:
  • This tool is relevant only for data in the ME subsystem.
  • Only one instance of the db-list tool can be executed at a given time. The tool cannot be executed in case a similar tool (for example, db-backup or db-restore tool) is currently running on the admincli pod.

  • The data is spread over multiple Aerospike sets, related by club-id or other pointers.

    Note that data inconsistencies may be present in the dumped data due to database updates taking place from provisioning or call processing transactions while this db-list dump it being performed. The amount of inconsistencies will be higher if db-list is run during peak traffic periods.

    For instance, to dump all data related to a device, the device is read from one set and then its custom data, aggregate view instances, and bucket instances among others are read from different set. If it takes x amount of time (usually in milliseconds for each entity) to dump the same data then, for instance, if any provisioning activity or 4G or 5G call or audit are ongoing during the same time frame then data dumped might be inconsistent. The case may arise where subscription id is read from device data and by the time tool tries to read the subscription object, it may already have been deleted from the database due to any activity mentioned above.

  • The db-list tool always starts from the beginning and there is no option to pause or resume a previous tool invocation. This means that, if the db-list tool is interrupted due to any reason or exception, when the tool is executed next time, it starts from the beginning. The data files created before interruption need to be manually deleted from the system.
  • Edigest is dumped with each <entity> option.
    Note:
    Edigest is a primary key that uniquely identifies a record in the Aerospike database within a given namespace. Primary key is used in application, while the digest is used for addressing the record in the database.
  • In case of dump taken from a system having a larger data size provisioned, for example millions of devices with all associated charging data, then these options in db-list tool are helpful:
    • t : number of threads
    • q : worker queue capacity (it is fixed to 100k records at present)
    • r : JVM heap size (it is 2G at present)

Syntax

db-list <see table for available options>

None of the options mentioned in this command are mandatory.

The tool can be launched without any options. If this occurs, then it uses the default value for each option.

Option Description Option arguments
No arguments Displays usage or help information.  
-h or -? Displays usage or help information.  
-l <entity> Displays the type of entity for which you need records listed. Only one entity at a time can be added in this argument. For example, -l <devices>

Default value of this option is -l devices.

The -l <entity> option of the db-list tool has these output options:

  • -l devices: Lists the devices in the InfrequentlyChangingData set along with the custom data, device subscriptions, subscription counter, associated bucket instances, threshold instances, account balance data, cap instances, device counters, Aggregate View (AV), and DPU present in the DynamicData set.

    Edigest is present for device, custom data, subscription, subscription counter, device counter, AV instance, DPU, account balance, and for bucket instance at CS level.

  • -l accounts: Lists the accounts along with the account administrator present in InfrequentlyChangingData and custom data attributes of the account present in the DynamicData set.

  • -l users: Lists the users along with the custom data attributes of the user present in the InfrequentlyChangingData and DynamicData set respectively.

    Edigest is present for user and custom data associated with account.

  • -l groups: Lists the groups in the InfrequentlyChangingData set along with the custom data, group subscriptions, associated bucket instances, subscription counter, threshold instances, cap instances, group and device_group counters, aggregate view, and account balance data present in the DynamicData set.

    Edigest is present for group, custom data, subscription, subscription counter, group counter, AV instance, device group counter, account balance, and for bucket instance at CS level.

  • -l sysessions: Lists only the sessions of Sy type present in the SessionContainer set.
  • -l ucs: Lists all the records present in the UsageControlService set.
  • -l allsubscriberdata: Lists all the subscriber entities data (which includes, devices, groups, users, and accounts).
  • -l gysessions: Lists all the Gy sessions present in the DynamicData and TerminatedSessions sets.
  • -l imssessions: Lists all the IMS sessions present in the DynamicData and TerminatedSessions sets.
  • -l ratingsessions: Lists all the rating sessions present in the DynamicData set.
  • -l sessions: Lists all the charging sessions (Gy, rating, IMS, and Sy) present in the DynamicData and TerminatedSessions sets.
  • -l CDR: Lists all of the CDR.
  • -l allprofiles: Creates a job for all profiles and dumps all device details, including account details that are present under subscriptions. This option is not available on SM systems. The only other supported option you can use with -l allprofiles is -o <dir>. The tool provides:
    • Database dump attributes matching to response attributes/structure of getData/Device APICombined JSON with device and account information.
    • Clear identification/mapping of ENUM attributes/values.
    • Clear mapping of composite information – mapping with getData/Device API
    • No Excel sheet CSV format data is dumped with this option.
    • When filtering of attributes from JSON Dump using JMS_PATH, only data dump is received.
    Note:
    Consult Nokia support before you use this option as part of a scheduled job or on large data sets (~1M or above). Due to the resource-heavy nature of the db-list tool with -allprofiles, do not run the tool with this option multiple times in parallel.
  • -l allprofiles_getstatus:<Job-id>: Displays the status of the allprofiles job such as Success, In-progress, or failed. For example, if the job ID is 1234 the syntax should be -l allprofiles_getstatus:1234. This option is not available on SM systems.
  • -l allprofiles_delete:<Job-id>: Aborts/deletes the allprofiles job including the dumped data. For example, if the job ID is 1234 the syntax should be -l allprofiles_delete:1234. This option is not available on SM systems.
-i <entity> Look for inconsistencies in the selected entity type and report on any issues that are found. The valid entity values are:
  • devices
  • groups
  • accounts
  • users
  • allsubscribers
  • sessions, or
  • the complete path to a schema file.

When providing a schema file path, provide the complete path to the schema XML file. The tool checks inconsistency of all the objects given in the file and then validates with the corresponding schema.
-a <report|heal> Action to be taken when an inconsistency is discovered and reported with the -i option. If heal is specified for sessions, then in addition to just reporting the inconsistencies, an attempt is made to heal the data where feasible. Healing actions are then also reported.
Notice:
Consult Nokia support before you use this option.
 The default is report.
-n <namespace|all> The Aerospike namespace to look in for the records.

Default value is the primary partition name of the local system.

The namespace should be a partitioned namespace. For example, the namespaces starting with the spspd prefix.

If all is specified, then the tool lists from all the partitioned namespaces.
-m <max|all> The maximum number of records to be listed.

Default value is all.

If all is specified, then the tool does not stop until it has processed all of the records.
-o <dir>

The output directory where the listed records are placed.

In the specified directory, the tool creates a new directory bearing the name of the selected object type and the current timestamp.
  • Ensure that you have sufficient disk space in the file system mount where the output directory is located.
  • Ensure that the user used to execute the tool has write permissions on the specified output directory.
 Default value is /opt/tpa/dump/.
-t <nThreads> Displays the number of processing threads.

Default value is equal to half the number of CPUs or VCPUs on the node.

A high number of processing threads increase the CPU utilization of the node. On the other hand, a lower processing thread count may engage Aerospike query threads for longer durations.

Nokia recommends you to select the number of processing threads nearest to the number of CPUs.
-f <nFiles> Displays the number of out files.

Default value is equal to the number of processing threads.

Nokia recommends you to keep the number of out files the same as the number of processing threads.
-q <nCapacity> Displays the capacity of queue in thread pool. Default value is 1000.
-r <heap> Displays the heap size (in MB) for database dump process. This option is set depending upon the available RAM.
-k <file_path>

The complete path for clubIds/CSV input file.

For information on subscriber healing, see Subscriber healing and orphan lookup tables audit.

For session inconsistency, CSV record format should be: <ClubId>, <DeviceId>, <E164>, <IMSI>, <SessionType>, <SessionId>. For each CSV entry, the club id is validated first, and if it does not match or is not present, then the next option provided in CSV is used. Session type is mandatory for session id in CSV.
  • Multiple options can be provided as input in CSV. The option that is validated first is considered and rest are ignored.
  • For each line, the first <ClubId> is considered and if it does not exist, then the next input is considered (that is, <DeviceId>), and so on.
  • If <DeviceId>, <E164>, <IMSI> or <SessionId> are not present in the database (lookup tables), then there is no inconsistency. In this case, run subscriber inconsistency process first.
  • If healing is required on the basis of IMSI, then the input parameter should be: , , ,<IMSI>.
  • If healing is required on the basis of session, then the input parameter should be: , , , , GY, <SessionId>
  • For each <Session Id>, a <SessionType> is mandatory. Possible session types are SY, GY, IMS (case sensitive).
  • For each input parameter in CSV file, NCC checks and heals all inconsistencies associated with the Club Id.
-j

Ensures that output does not contain any keys corresponding to id values of certain objects, and the format of these objects is changed to array of objects.

It formats the JSON output as array of objects instead of object of objects.

This should be used with the other arguments:

[-l ] [-n ] [-m ].
-d <duration>

Duration in seconds. To be used with -l sessions to request a dump of all sessions older than the given duration. This can help identify old or stale sessions.

 The default is 24h.

If the number of reservation maps in acctBalances, bcktInsts, and entityCtrInsts is greater than or equal to Maximum Number of Reservations (100), then the entity is skipped and the status is set to MAX_NUM_RESERVATION_FOUND. Database healing is not performed for such cases. This is applicable with these options:

  • ./db-list -i accounts -a heal -k <File with Club ID of account>
  • ./db-list -i devices -a heal -k <File with Club ID of device>
