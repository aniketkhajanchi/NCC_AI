Restore non-replicated Aerospike data from Backup and Recovery

You must restore the Aerospike data on the recovering system, SM4. The non-replicated data is restored from the latest Backup and Recovery backup and the replicated data is restored from another healthy SM.

Truncate the Aerospike database namespace

This command clears all mandatory partitions. For IG (dsclocal, dscglobal, dsc, spspd) and for SM (dsc and dsclocal).

kubectl -it exec <admincli> -c <container> -n <namespace> -- /opt/tpa/bin/disaster-recovery-helper.py -t all
# Type y or yes and press enter when it ask for confirmation “Do you want to continue?: y|yes|n|no”

Restore non-replicated Aerospike data from Backup and Recovery

## To get the database backup from Backup and Recovery, use this command from the admincli pod, where $RELEASENAME is the release name of admincli in your installation:
curl -vkL --header 'Content-Type: multipart/form-data' --header 'Accept: application/json' -X POST --header 'Content-Type: application/x-www-form-urlencoded' -d 'backup_id=&volume=&object_type=&object_name=' http://cbur-master-cbur.$NAMESPACE.svc.cluster.local:80/v2/helm/release/restore/$NAMESPACE/$RELEASENAME?helm_version=3
  
## By default, Backup and Recovery master is installed with Celery/Redis to support asynchronous functions, so the restore Rest API calls will return a taskId
## which can be used to query the status of the restore with another Rest API call, like in the following command, that has an example taskId:
curl -v --header 'Accept: application/json' -X GET http://cbur-master-cbur.$NAMESPACE.svc.cluster.local:80/v2/task/7b59914d-f521-41552b81b-765a45f85a42

## When the restore is complete, confirm the backup has been copied to admincli. Still in the admincli pod on SM4 do:
ls -al /opt/tpa/data/db-backup
 
## In the admincli pod of SM4 issue the following command to restore the dsclocal namespace of the database
/opt/tpa/bin/db-restore -d /opt/tpa/data/db-backup -n dsclocal -f
