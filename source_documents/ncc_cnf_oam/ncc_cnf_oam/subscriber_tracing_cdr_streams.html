Subscriber tracing

In some cases, to debug charging errors or missing CDRs, an operator may want to track what is happening for a given subscriber in the system. For this purpose, the system provides record level tracing functionality. Records generated by different NCC modules may contain fields which indicate that tracing is enabled and the identifier to be included in the trace log messages generated.

In NCC generic record these fields are subscriberTracing (with value true/false) and subscriberTracingId. The tracing fields are configured by default in the Business logic stream (with node parameters SubscriberTracingFlagField and SubscriberTracingIdField). The tracing is, however, only triggered when both the configured fields are present in the input record and the field set in SubscriberTracingFlagField has value true (this value is case insensitive).

With the node parameters, you can also modify the field names that are used for enabling tracing and set the tracing identifier.

When a record contains the required fields and has tracing enabled, CDR streams generate a trace log message in the following format:

<TRACING_ID> <LOG LEVEL> <MESSAGE>

Example:

DEMODEVICE TRACE CDR file successfully distributed to /data/cdr.dat

Trace logs are generated by CDR streams as described in the following table.

Table: CDR streams generating subscriber trace logs
CDR stream Subscriber trace log is created when
Business logic stream
  • Output record has been created after mapping. The full record is logged in readable format as well as all output types it was sent to (FILE/KAFKA).
  • Output record was filtered out by distribution rules (no output configured).
  • Output record was filtered out by Business logic stream node parameter settings (disabled outputs).
  • Input record has been rejected. The full record is logged in readable format.
File output stream
  • Record has been distributed successfully. The remote file name and the path to the stored record is logged.
  • Record distribution has failed.
Kafka output stream
  • Record has been distributed to Kafka broker successfully.
  • Record distribution has failed and the record sent to Failure logs.
  • Record distribution to both Kafka broker and Failure logs has failed.
Log distributor stream
  • Failure log has been distributed successfully. The remote file name and the path to the stored log is logged.
  • Failure log distribution has failed.
Late rating business logic stream
  • Input record has been rejected. The full record is logged in readable format.
  • Output record has been created after mapping. The full record is logged in readable format.

You can find the trace logs in these locations:

  • In stream pods under path /opt/comptel/eventlink/base/tracelog
  • In Charging Gateway Function UI, using the Execution trace logs link for a specific stream
  • In OpenSearch Dashboard. For more information, see the Subscriber Manager Guide.
