Rates

Rate defines the amount of currency to be charged for each unit of consumption by a call. For example, 0.05 United States dollars is charged for 1 SMS.

The rate entity defines the rating conversion parameters and the type of the resource allocated or reported. Rate is applied to a single unit type or type of consumption resource such as time, volume, and unit. It can define the type of resource or consumption for which it applies.

It is recommended not to change the rate/tax in between calls when the session is ongoing.

The rate entity allows operators to define the following:

  • Define granularity of rating

    For example, the rate is expressed as 5 KB for 0.25 United States dollars. However, beyond this rate, it is important to identify the granularity of the charge. The granularity may be of a different quantity than that expressed in the rate. In this example, the granularity may be 512 bytes or 1/10 of 5 KB.

  • Define a step rate.

    For example, 0.05 United States dollars for the first 100 MB, and 0.03 United States dollars after that.

  • Perform one of the following actions for step-rate entity:

    • Define the step-rate entry duration in the units being rated.

    • Define unlimited number of steps in one rate.

Examples

Example 1: Normal cost

Rate to define a normal step type to charge 1 Euro for every 10 MB usage

  1. For first 100 MB consumption, 1 Euro is charged per 10 MB usage.

  2. For the remaining part of the call, 0.5 Euro is charged per 10 MB usage.

Example 2: Fixed cost

Rate to define a normal step type to charge 0.025 Euro for every 10 MB usage for first 100 MB and then deduct Fixed cost = 10 Euro for 50 MB; remaining call is charged with 0.50 Euro for every 10 MB usage

  1. For first 100 MB consumption, 0.25 Euro is charged per 10 MB usage.

  2. Fixed cost of 10 Euro is deducted for 50 MB.

  3. For the remaining part of the call, 0.50 Euro is charged again per 10 MB usage.

Voice consumption charged from a Bucket of Unit type

Voice consumption can be charged from a Bucket of Unit type. Operators can use bucket of generic units (that is, Unit type) to be charged for Voice related usage.

Note:
For NCC rating computation, the unit type of the resource should be same as per requested service unit type (RSU). Only Bucket of Unit type can be charged for the seconds reported in Voice Call, where the requested service unit type is TIME.

Example

Scenario

Consider that a Device has two subscriptions. The higher priority subscription has a charging from Bucket with 32 generic charging units, with Rate of 15 cost/min, and granularity of 60 sec and the lower priority Charging Service (CS) has a Charging from main balance with Rate 0.09 cost/min and granularity of 15 sec. The Account Balance is set to 10.

Execution

Execute an IMS call for a duration of 130 sec.

Expected behavior

  • For the 120 seconds the consumption should be done from the Bucket and 30 generic convertible units should be consumed from it (as the Rate is of 15 units/sec with granularity of 60 sec), leaving 2 generic units in the bucket.
  • For the remaining 10 seconds, it should be covered from the Account and as the granularity of the Rate is 15 sec. Therefore (15*0.09€/60 = 0.025 €) 0.03 € is deducted.

Versioning for rate

Versioning of a charging logic is available for Rate component update. When publishing a new version of charging logic (with Rate component values update), the new version impacts all the relevant subscriptions (per charging logic) with corresponding new Rate values.

When Bundle and CL versioning is ON:

The Rate component can be versioned by overriding the values and save them in CL versioned entity. The Rate component in the charging logic can be:

  • Global Override for Rate - No - Default value (currently). Updating the Rate component (not in CL context) values impact all the subscriptions immediately (not related to version type).
  • Global Override for Rate - Yes - The CL /Rate component can be editable as part of versioning and impacts the relevant subscriptions (per version type).
    Note:
    The Rate component can be replaced in the CL (this is considered per version type).
  • Supporting all types of versioning: Immediately, Next BCR, Protect, Patch versioning and Emergency fix (for Next BCR and protect)
  • Supporting CL / Rate versioning for late calls
  • Supporting CL / Rate versioning for rerating
  • Any rate entity can be changed
  • Keeping the current rating functionality (no change)

When Bundle and CL versioning is OFF (consider backward compatibility):

  • Any change in the Rate component (not in the CL) impacts the existing subscriptions immediately.

Versioning during a call

Rate component can be versioned by overriding the values and saving them in charging logic versioned entity as an embedded object instead of keeping it as reference.

  • On Version change and version Type is Immediate → follow charging logic versioning
  • On Version change and version Type is BCR→ follow charging logic versioning
  • On Version change and version Type is Protected → follow charging logic versioning

Scenarios: Versioning for rate

Versioning OFF

Adding a Rate component (not editable) to a CL:
  • Create a Rate component ($5/1MB, Step Size 5MB, Fixed Fee $10)
  • Create a Charging Logic
  • Add the Rate component to the CL - not editable
    Expected results: The rate in the CL cannot be modified.

Versioning ON

Adding a Rate component (editable) to a CL:

  • Create a Rate component, Rate1 ($5/1MB, Step Size 5MB, Fixed Fee $10)
  • Create a Charging Logic (charging logic v1.0)
  • Add the Rate component to the CL - now editable
    Expected results: Values for rate are editable, and can be updated.
    Change the Rate component (outside the CL) - it is editable and impacting the existing subscriptions.
  • Update Rate1 (with charging logic v2.0) component ($5/1MB, Step Size 10MB, Fixed Fee $15, Charging Granularity 0)
  • Publish Charging logic v2.0 with Rate 1 updated values. Charging logic v1.0 becomes obsolete, and Charging logic v2.0 , Active.
