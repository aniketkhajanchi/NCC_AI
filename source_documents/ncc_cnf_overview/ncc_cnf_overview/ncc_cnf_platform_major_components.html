NCC base components

The NCC architecture consists of several groups of microservices, including:

  • Ingress and Load Balancers

    These services terminate the incoming traffic and load balance it to the appropriate pod instances within the NCC.

  • Service Manager

    These microservices are unique for the SM subsystem. These microservices terminate the external SOAP and REST APIs and manage the data provisioning for all of the NCC systems in the network. The SM subsystem provides a GUI for manual data viewing and updates, and also includes a User and Role Management component for managing the authentication and authorization of GUI and API users.

  • SS7 Call Handling

    These microservices provide the protocol handling and business logic associated with SS7-related services in NCC.

  • Internal Queues and Pub/Sub

    These provide mechanisms for NCC services to publish events and other queued information to a variety of NCC services that are subscribed. Kafka is used for forwarding CDR requests from the charging application to the Charging Gateway Function (CGF). Kafka is also used for a variety of publish / subscribe (pub/sub) services. For example, the database publishes data change notifications to any clients who have subscribed. These microservices also provide a timed events manager and inform subscribers when given events are triggered.

  • Subscriber Database Services

    This group of microservices provides a variety of services related to the subscriber data that is managed by the charging application. This includes the NoSQL database and its replication across other NCC systems in the network deployment. It also includes the services for ensuring data integrity, including serializing transactions through an active home ME subsystem for each particular subscriber and routing the messages to the correct ME subsystem in the network. Data integrity is also ensured by a lock and transaction manager to mange the simultaneous requests for a specific database record, with capabilities for rollback of failed transactions.

  • System Configuration and Management

    These microservices provide system level configuration, service discovery, and monitoring. This includes network heartbeat communication with other NCC systems in the network deployment to construct an accurate view of NCC system status and avoidance of scenarios such as split-brain behavior. This also includes system-level local monitoring that considers the composite view of local service instances to determine the overall health of the local system, what to communicate in the heartbeat, and whether the system is healthy to be in-service for offered traffic. These services also provide a pod for running command-line interface tools (via kubectl). Finally, system backup and restore is managed via these services.

  • Charging Gateway Function

    These microservices manage all of the CDRs in NCC. For more information on the CDRs, see NCC Charging Gateway Function (CGF [DR]) and CDRs.

  • Telemetry

    NCC Telemetry is an optional component that provides all the services for logs, alarms, and metrics. For information on NCC Telemetry, see NCC Telemetry components.
