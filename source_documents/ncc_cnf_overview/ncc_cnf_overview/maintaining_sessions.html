Maintaining sessions

External requests from the core network to SS7 are handled by SS7FEP pods. These SS7FEP pods work in Active-Active Modes with SS7 link redundancy among them.

As soon as the external core network detects failure of SCTP association, the external core network starts diverting all messages to its other available ACTP association with the other SS7FEP pod.

SS7FEP hands over SS7 messages to SS7BEP pods. SS7 maintains a set of active BEPs. When a request comes to the front end process IOH, that request is sent to one of active BEPs. This maintains session throughout multiple requests. In case any of the active BEP fails, then all the ongoing TCAP sessions associated with the failed BEP are dropped. However, new TCAP message are processed by the available active BEP.

SS7RTCC pods maintain SS7 session details in a Mariadb table. When a SS7RTCC pod fails, the SS7BEP pod hands over request to other available SS7RTCC pods. These SS7RTCC pods retrieve session details from Mariadb to process transactions seamlessly.

The SS7 USSDGW session state is stored locally in the MariaDB for SS7 session data. If a USSD pod fails, subsequent transactions are sent to a different USSD pod which retrieves the transaction state from MariaDB and processes the transaction. TCP retry only. Error sent to SS7 peer.
