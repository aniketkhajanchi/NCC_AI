System recovery from XDR lag

After parallel database pods restart due to high memory, multiple partitions can go into a dead state. Perform these steps to recover the partitions.

  1. Run the db-roster tool from the admincli pod to check for dead partitions and Aerospike Cluster Status. 
    ./db-roster -v
  2. If dead partitions are found:
    1. Run this command on the SM/ME systems (run on both for IG systems) to set the MOOS flag: 
      kubectl -n $NAMESPACE -it exec <admincli-pod> – /opt/tpa/bin/configure-manual-oos -d <SM/ME>
  3. If the Aerospike Cluster Status is DOWN_DRR, follow the full Disaster Recovery procedure.
  4. If the Aerospike Cluster Status is not DOWN_DRR, perform these steps:
    1. Run this command for all namespaces that show dead partitions: 
      /opt/aerospike/bin/asadm/asadm -e "manage revive ns <namespaceName>"
    2. Run the recluster command :
      /opt/aerospike/bin/asadm/asadm -e "manage recluster"
    3. Restart all application pods before you unset the MOOS flag.

      Other than database pods, all other application pods (for example: Charging, Provisioning, Notification, CM, IOHD, SMapp, and other MS service pods) must be restarted.

    4. Run this command to unset the MOOS flag. 
      kubectl -it exec <admincli> -c <container> -n <namespace> –/opt/tpa/bin/configure-manual-oos -e <ME/SM>
