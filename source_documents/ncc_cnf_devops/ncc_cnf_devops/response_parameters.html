Response parameters

Here are described the HTTP status codes and response body parameters received for the alternative REST API calls.

HTTP status

Table: HTTP statuses for REST API call responses
HTTP code Description
200 Successful. The request was successfully processed.
401 The user credentials are incorrect.
403 Forbidden. The user is not permitted or does not have the appropriate role.
404 Not found. The requested item was not found.
405 Invalid method. The functionality is disabled.
409

Conflict. The requested item is not unique.

Note:

For a REST API call cancelling a test suite, this status code means:

Conflict. The job cannot be cancelled because it has already completed.
500 Internal server error. An unexpected error occurred while processing the request.

Response body parameters

Response body parameters are different for the three possible high-level actions (starting, monitoring and cancelling a test suite) and for creating a test suite artifact. Therefore, they are presented in separate tables.

Table: Response body parameters for starting, monitoring and cancelling a test suite
Name Description
name The name of the test suite. Possible values: NEW_FEATURES, REGRESSION, PERFORMANCE
id

A unique identifier of the test suite being executed (for example, 0578a6f6).

Additionally:

  • For REST API calls starting a test suite and monitoring a test suite:

    A NOT_BUILT result can have an id of -1 or the id of the job that has not started yet.

  • For a REST API call starting a test suite:

    A DUPLICATE result will have an id of -1.
result The result for the request. Possible values: SUCCESS, FAILURE, ABORT, NOT_BUILT, DUPLICATE
error The error message on FAILURE.
vendor The vendor name.
product The product name.
release The release number of the vendor software.
estimatedDuration An estimated duration of the test suite in milliseconds.
duration The duration of the test suite execution in milliseconds.
timestamp The time in milliseconds since Jan 1 1970 (UTC).
tests The total number of test cases in the suite (not steps but individual test cases).
errors The number of test cases that have resulted in errors (unexpected problems).
failures The number of test cases that have failed (expected problems).
attempts The number of times this test suite has retried in case of failure.
Table: Response body parameters for creating a test suite artifact
Name Description
name The name of the test suite. Possible values: NEW_FEATURES, REGRESSION, PERFORMANCE
id A unique identifier of the test suite being executed (for example, 0578a6f6)
property:vendor.name The vendor name.
property:vendor.product The product name.
property:vendor.release The release number of the vendor software
time The time taken (in seconds) to execute the tests in the suite.
timestamp The time when the test was executed, in ISO 8601 format.
tests The total number of test cases in the test suite.
failures

The total number of tests (in the test suite) that failed. A failure is a test where the code has explicitly failed in a testing mechanism, for example, assertEquals.

The test suite is considered a SUCCESS only if both failures and errors have value 0. Otherwise, the test suite is considered a FAILURE.
errors

The total number of tests (in the test suite) that had an error. This is a test case that had an unanticipated problem, for example, an unchecked throwable, or a problem with the implementation of the test.

The test suite is considered a SUCCESS only if both failures and errors have value 0. Otherwise, the test suite is considered a FAILURE.
attempts

The number of times the test suite has retried in case of failure.
