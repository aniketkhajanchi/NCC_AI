Throttling profile

NCC supports limiting the rate at which requests can ingress and egress NCC. The throttling parameters are configured through profiles.

There are four types of throttling profiles, in order of execution:

  • Inbound, for example used to throttle requests that originate from a particular client origin host or realm or from a Diameter application and command combination. This is used at system level.

  • Ingress Peer, for example used to throttle requests arriving via a particular directly connected peer.

  • Egress Peer, for example used to throttle requests routed by NCC through a directly connected peer.

  • Outbound, for example used to throttle requests destined to a destination host or realm. This is used at system level.

All profile types support adding further criteria (for example, application and command) to focus throttling on more specific scenarios.

A request can be throttled by invoking a discard or reject action on the request.

The criteria and actions of the profiles are configured as message filters. For more information about message filters, see the Operations, Administration, and Maintenance Guide.

Configuration

To configure the throttling profile, in SM GUI navigate to Connectivity → ☰ → Throttling. Select PROFILES tab. Click +, and select Inbound Throttling Profile or Outbound Throttling Profile and configure the parameters as described in the following table:

Note:
The message rate for the throttling is configured as based on the aggregate traffic for all the charging pods. Each charging pod supports a message rate that is the configured rate divided by the number of active charging pods. For example, when there are five charging pods sharing the processing load, 10,000 requests per second is applied as 10000/5=2000 requests per second on each charging pod.

You can configure either the Max reqs per interval or the Interval Size, or both, to increase or decrease the overall rate. For example, if you want to throttle an ingress peer at 100 messages per second, you could configure the following:

  • Max reqs per interval = 10

  • Interval size = 100 ms

10 messages per interval /interval size of 0.10 = 100 messages/second

By reducing the Interval size, the same result can be achieved by changing the Max reqs per interval, as follows:

5 messages/50 ms Interval = 100 messages/second
