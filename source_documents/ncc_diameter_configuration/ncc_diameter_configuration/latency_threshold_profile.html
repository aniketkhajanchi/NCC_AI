Latency threshold profile

Latency is measured from the time the request is received by the Diameter application until the time the Diameter application sends the response. There is an application preference: "Diameter Request Timeout Duration(ms)" that checks when a request starts to be processed and whether the request has been sitting queued in NCC for too long. If the request has been in the queue too long, it is not processed since it is not possible to respond within the guaranteed response time. See the Global Configuration Guide for information about setting this application preference.

For the latency thresholds, the settings depend on the latency performance expected by the customer and the normal range of latencies that are being observed. You do not want the upper threshold to be too low or the system will end up in an overloaded state too frequently, resulting in subscribers not getting the service they expect. For example, a customer may want a typical response time of 50 milliseconds, and have a hard limit of say 1800 milliseconds (specified by the "Diameter Request Timeout Duration(ms)" application preference). Looking at the Metrics Viewer graphs of processing latencies that show how the system currently performs may give you a good indication of where to set the upper and lower threshold.

To configure a latency threshold profile, select PROFILES tab in the Overload Protection window. Click + and select Latency threshold profile. Configure the latency threshold profile parameters as described in the following table:
Table: Latency threshold profile parameters

Parameter

Description

Mandatory or Optional
Name Specify the profile name. Mandatory
Lower threshold Specify the threshold, in milliseconds, under which overload can be cleared. For latency, the threshold is specified in milliseconds. Mandatory
Upper threshold Specify the threshold to trigger a major alarm. Mandatory
Resource critical threshold Specify the threshold to trigger a resource critical alarm. Mandatory
