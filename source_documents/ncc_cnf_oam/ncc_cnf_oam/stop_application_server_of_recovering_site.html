Stop the recovering system application server

The application server on the recovering system must be stopped. This example shows the flow that is required to stop the application server.

## For pods not using the Horizontal Pod Autoscaler, you must determine the number of replicas before scaling down, so you can correctly
## scale them up once the database restore is complete. Use this command, where $CM_STATEFULSET is the name of the centralmanagemt statefulset in your deployment:
CM_REPLICAS=$( kubectl  -n $NAMESPACE get statefulset $CM_STATEFULSET -o jsonpath="{.status.replicas}")
IOHD_REPLICAS=$( kubectl  -n $NAMESPACE get statefulset $IOHD_STATEFULSET -o jsonpath="{.status.replicas}")
LTM_REPLICAS=$( kubectl  -n $NAMESPACE get statefulset $LTM_STATEFULSET -o jsonpath="{.status.replicas}")
     
## check the values have been saved correctly:
echo $CM_REPLICAS
echo $IOHD_REPLICAS
echo $LTM_REPLICAS
  
## Scale down the pods that would be affected by the Aerospike database restore operation.
## To stop the pods before launching db-restore script:
kubectl -n $NAMESPACE scale --replicas=0 deployment -l scaleForDbRestore=true
kubectl  -n $NAMESPACE scale --replicas=0 statefulset -l scaleForDbRestore=true
