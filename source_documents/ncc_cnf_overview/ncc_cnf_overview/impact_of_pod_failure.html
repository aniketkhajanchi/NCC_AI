Impact of pod failure

When a pod fails, Kubernetes liveness probes detect the failure and schedule a replacement pod.

Transactions in progress on a failed pod do not return a response.

Pods that are first to handle external requests from the service mesh return 503 Service Unavailable for HTTP, or for the IOHD diameter UNABLE_TO_DELIVER or UNABLE_TO_COMPLY.

If the active IOHD pod fails but before centralmanagement detects a failure, new transactions sent to the failed active pod do not return a response and the diameter peer times out.

If the active IOHD pod fails and there is no standby pod available, no response is returned and the diameter peer times out.

If the active Lock and Transaction Manager (LTM) pod fails and the standby LTM has not yet taken over, application pods that send an API request to LTM time out waiting for a response and return an HTTP error or diameter error.
