NCC dimensioning tool

Information from the Dimensioning Tool needs to be updated in the input yaml files for the number of pod instances and resources requirement.

The NCC Dimensioning Tool calculates several parameters that need to be entered into the CNF Input Generator Tool. The tool calculates the values of these parameters, based on input parameters like transaction rate, device count and subscriber count.

The NCC Dimensioning Tool also provides the following information which is important in planning the deployment.

  • NCC backup size: Backups should always be copied to remote location as well. This parameter gives an indication of the required space in the system to hold the back-ups. Default, we consider only one copy of back-up
  • Write block size and IOPS: The environment into which NCC is to be deployed must be capable of sustaining this number of IOPS with the specified block size, as a minimum
  • Write storage IO rates: The environment into which NCC is to be deployed must be capable of sustaining this storage IO rate, as a minimum

Charging inputs into the NCC dimensioning tool

Below inputs are required to dimension the system

  • Number of devices (subscribers)
  • Number of sessions (total of all session types)
  • Maximum RAM allowed per Aerospike pod instance
    • A cloud environment may specify, for example, that the largest allowable RAM is 112 GB. That limitation can be specified in the Dimensioning Tool, and the Dimensioning Tool will calculate the number of required Aerospike pod instances based on that limitation

  • Whether the deployment is IG or separate SM and ME

  • Whether or not CDR/EDRs are enabled
  • The CDR/EDR retention time, in hours
  • Gy/N40 TPS
  • Sy/N28 TPS
  • SS7 CPS (if required)
  • Numbers of: devices, users, accounts, groups, buckets, counters, subscriptions and custom data
  • Number of sessions
  • The CDR/EDR retention time, in hours
  • Provisioning TPS
  • Notification TPS
  • PLC Renewal Details
  • Data Encryption

The Dimensioning Tool inputs are part of input sheet and depending on deployment type the output will be shown in the respective output. Further, year wise configuration is available in the respective output sheet.

Dimensioning considerations

Below are the important considerations while doing NCC :

  • BTEL and other CSF components like CKEY, CBUR, etc are part of the dimensioning resource output.

  • istio-side cars dimensioning are part of the main pods and included into the dimensioning output.

  • Other istio service mesh dimensioning is not part of the dimensioning.

  • Kubernetes cluster dimensioning like Controller, deployer and Edge nodes are not part of the dimensioning. For more information, contact NCC customer support to add them into overall resource calculations.

Scaling

CNF NCC supports auto scaling provided by Kubernetes known as Horizontal Pod Autoscaling. Below pods are part of Horizontal Pod Auto-Scaling

  • Charging

  • Notification

  • Provisioning

  • Service Manager

  • xdrproxy

You can configure the maximum and minimum for these pods based on the requirement and overall dimensioning.

In service upgrade

NCC requires In Service Upgrade and for ISU operation to be completed successfully. There is an expectation that enough resources exist in the system to allow for the creation, initialization and execution of the new pod at a time when the old pod is still running and consuming all of its resources. See the Installation and Upgrade Guide for more information.
