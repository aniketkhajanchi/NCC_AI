Helm upgrade to put the recovered system back in service

Once all of the data is recovered, generate new overrides using the same input-generator, but this time specifying Deployment Type as in-service upgrade and Enable XDR traffic as true.

This step creates the Kubernetes objects that are needed by the system to communicate and replicate data.

Perform the Helm upgrade and when it is completed verify the status of the replication for:

  • Web SSO data, as described in Verify MariaDB replication.
  • Aerospike data, as described in Verify that XDR replication status is current. The XDR replication between the recovered system and the other systems should take care of updating the data on the recovered system to the current state. This might take a while, but eventually the replication should synch-up. If this does not happen, there is the option to perform the procedure described in Aerospike data recovery, to manually perform a recover from an healthy system to the recovered system.

You should also verify the SM configuration as suggested in Verify/update SM configuration and the status of CGF (DR) dr-ui pods, as explained in Verify the status of CGF (DR) dr-ui pods.
