xdr-status

Description

This command displays Aerospike Cross Datacenter Replication (XDR) metrics along with both local and remote system's XDR status.

The tool creates status output for each of the systems and includes status information with XDR replication status (local site towards sites): and XDR replication status (remote sites toward site):.

The number of systems listed is always equal to the number of geo-redundant systems of the same site type that have been configured under the topology section of the Helm chart as described in the Installation and Upgrade Guide.

The tool reports the worst-case among the XDR metrics reported by each Aerospike pod of a given system.

All metrics are for information purpose, except for lag. Only lag is used to determine whether XDR is falling behind or not.

Note:
If the XDR SyncUpState of a remote data center (DC) is BEHIND due to a particular namespace(s) and for some reason it is needed to make it CURRENT so that monitoring isn't affected, then an SDC key /<k8sNameSpace>/services/aerospike/xdr/excludedNamespaces must be set to avoid the particular namespace(s) from being used in the XDR SyncUpState calculation for a particular remote site. The remaining namespace(s), after excluding the namespace(s) set though the SDC key, are used to evaluate lag for the particular remote site. The lag is then calculated for each remaining namespace(s) and the maximum lag obtained is used as the lag for that particular remote site to get the XDR SyncUPState. This update is seen in the output of xdr-status under XDR replication status (local site towards remote sites):.

Run this command to obtain the namespaces on the local server for a DC: 

/opt/tpa/aerospike/bin/asinfo -v "get-config:context=xdr;dc=<DC>"

Example for finding namespace for DC = spssite8:

/opt/tpa/aerospike/bin/asinfo -v "get-config:context=xdr;dc=spssite8"
                    
node-address-port=me006.hpgten26cnf.ih.lucent.com:3009;namespaces=dscglobal,spspd1,spspd3;auth-mode=internal;auth-password-file=null;auth-user=null;connector=true;
max-recoveries-interleaved=0;max-used-service-threads=0;period-ms=100;tls-name=null;use-alternate-access-address=false

From the output, it can be seen that namespaces=dscglobal,spspd1,spspd3; means dscglobal, spspd1, and spspd3 are namespaces for spssite8 that are configured on this server.

Run this command to obtain the lag per namespace:

/opt/tpa/aerospike/bin/asinfo -v "get-stats:context=xdr;dc=<DC>;namespace=<NamespaceOfDC>;"

Example for finding the lag of DC = spssite8 and NamespaceOfDC = spspd1 (Checking for each namespace among namespaces dscglobal, spspd1, and spspd3):

/opt/tpa/aerospike/bin/asinfo -v "get-stats:context=xdr;dc=spssite8;namespace=spspd1;"

lag=3727;in_queue=50;in_progress=1630;success=0;abandoned=0;not_found=0;filtered_out=0;retry_no_node=0;
retry_conn_reset=0;retry_dest=45174332;recoveries=2066;recoveries_pending=0;hot_keys=1117;
uncompressed_pct=0.000;compression_ratio=1.000;throughput=0

From the output, it can be seen that lag=3727; which means that the local server is unable to perform XDR replication of namespace spspd1 on spssite8 since 3727 seconds causes the XDR status for spssite8 to be BEHIND (local site towards remote sites):. Similarly, for dscglobal and spspd3 the lag was zero. It means that for spssite8, except spspd1, the remaining namespaces dscglobal and spspd3 can be used to decide XDR SyncUpState. You must exclude the spspd1 namespace to make the XDR status CURRENT by running this command:

Command: sdcctl set /<k8sNameSpace>/services/aerospike/xdr/excludedNamespaces
\"<DC1>=<Namespaces1 separated by comma>\;<NameOfDC2>=<Namespaces2 separated by comma>\;<NameOfDC3>=<Namespaces3 separated by comma>\"
Output: "<DC1>=<Namespaces separated by comma>;<NameOfDC2>=<Namespaces separated by comma>;<NameOfDC3>=<Namespaces separated by comma>"

The output indicates:

In DC1, the namespaces1 are to be excluded and remaining namespaces will be used to find lag for DC1.

In DC2, the namespaces2 are to be excluded and remaining namespaces will be used to find lag for DC2.

In DC3, the namespaces3 are to be excluded and remaining namespaces will be used to find lag for DC3.

A comma (,) is used only when there is more than one namespace to be excluded for any remote site.

A semicolon (;) is used only when there is more than one data center/remote site whose namespace needs to be excluded.

You must escape double quotes (") and semicolons (;).

When the DC and Namespace count is different then,

# Add SDC Key (DC = 1, Namespace = 1 [Comma(,) : NO & Semicolon(;) : NO])
sdcctl set /<k8sNameSpace>/services/aerospike/xdr/excludedNamespaces \"spssite8=spspd1\"
# Add SDC Key (DC = 1, Namespace = Multiple, [Comma(,) : YES & Semicolon(;) : NO])
sdcctl set /<k8sNameSpace>/services/aerospike/xdr/excludedNamespaces \"spssite8=spspd1,spspd3,dscglobal\"
# Add SDC Key (DC = Multiple, Namespace =1 [Comma(,) : NO & Semicolon(;) : YES])
sdcctl set /<k8sNameSpace>/services/aerospike/xdr/excludedNamespaces \"spssite8=spspd1\;spssite4=spspd2\;spssite7=spspd3\"
# Add SDC Key (DC = Multiple, Namespace = Multiple , [Comma(,) : YES & Semicolon(;) : YES])
sdcctl set /<k8sNameSpace>/services/aerospike/xdr/excludedNamespaces \"spssite8=spspd1\;spssite4=spspd2,dscglobal\"

Example:

sdcctl set naperilg28vzwcchf-y-nk-me-002/services/aerospike/xdr/excludedNamespaces
\"spssite8=spspd1\;spssite9=spspd2\;spssite3=spspd1,spspd2\"
                    
"spssite8=spspd1;spssite9=spspd2;spssite3=spspd1,spspd2"

The output means:

For spssite8, the namespace spspd1 will be excluded and remaining namespaces of spssite8 site on local site will be used to get lag for spssite8.

For spssite9, the namespace spspd2 will be excluded and remaining namespaces of spssite9 site on local site will be used to get lag for spssite9.

For spssite3, the namespaces spspd1 and spspd2 will be excluded and the remaining namespaces of spssite3 site on local site will be used to get lag for spssite3.

After setting the SDC key, the XDR status changes to CURRENT from BEHIND for spssite8, spssite9 and spssite3 in the output section XDR replication status (local site towards remote sites):.

When it is no longer required to exclude any namespace from the XDR lag calculation, run this command to remove the SDC key: 
sdcctl rm /<k8sNameSpace>/services/aerospike/xdr/excludedNamespaces

Syntax

xdr-status [-d <OOS-DRR>] [-w <OOS-WFR>] [-h]
Option Description
No arguments Display XDR metrics and status for the local and remote systems.
-h This help output.
-d <OOS-DRR> The OOS-DRR override value, it can be true or false.
-w <OOS-WFR> The OOS-WFR override value, it can be true or false.
