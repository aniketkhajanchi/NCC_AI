System recovery from OOS-DRR status

When NCC is down for a long duration, the tombstones on the healthy NCC can build up to the point that the Aerospike memory is nearly exhausted. This condition causes Aerospike to enter a stop-write mode.

Based on the memory and tomb-raider-eligible-age configuration of the healthy system, it is possible that the healthy system status changes from unhealthy to OVERFLOWED. The unhealthy system shows a state of OOS-DRR in the output of the xdr-status tool.

To recover from the OOS-DRR state, you must perform a disaster recovery procedure on the unhealthy system. For the disaster recovery procedure, see Disaster recovery (DR) and Disaster recovery for SM systems.

The disaster recovery procedure does not remove the OOS_DRR status. To update the status of the unhealthy system, run these commands.

# Verify on the unhealthy system the XDR replication status (remote sites toward local system) should be in CURRENT not OVERFLOWED for all sites.
# Verify on the remaining sites XDR replication status (local system towards remote sites) should be CURRENT not OVERFLOWED for unhealthy system.
kubectl -it exec <admincli> -c <container> -n <namespace> -- /opt/tpa/bin/xdr-status

# If all sites are in CURRENT or BEHIND, run this command:
kubectl -it exec <admincli> -c <container> -n <namespace> -- /opt/tpa/bin/xdr-status -d true

# Verify system status changes from OOS-DRR to CURRENT.
kubectl -it exec <admincli> -c <container> -n <namespace> -- /opt/tpa/bin/xdr-status

# When the internal service state changes to CURRENT, run this command.
kubectl -it exec <admincli> -c <container> -n <namespace> -- /opt/tpa/bin/xdr-status -d false
