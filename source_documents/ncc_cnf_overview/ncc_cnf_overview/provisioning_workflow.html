Provisioning workflow

All NCC provisioning is done by using the SM subsystem, whether it is through the GUI or through the APIs. The SM subsystem manages the mapping of which subscribers/devices are homed on which ME subsystems, based on various algorithms. This includes keeping groups together on the same ME subsystem (except for large enterprises), a least-loaded ME algorithm for new subscriber groups.

The SM subsystem interacts with the ME subsystems through SOAP messages to perform these tasks:
  • Accept and process the provisioning requests from customers’ upstream CRM systems.
    Note:
    NCC does not prevent the upstream system from provisioning to either of the SM subsystems in a network deployment. However, data conflicts are possible if related APIs are sent to opposite SM subsystems. Nokia recommends that back-to-back transactions for related devices, groups, or accounts be sent to the same SM subsystem to avoid race conditions.
  • Perform the SM GUI actions through the REST API gateway on the SM subsystem.
  • Accept and process the SOAP provisioning requests that include actions for service data, subscriber data, and subscription data. Charging is responsible for implementing the requests including execution of any business logic that might be required.

These are the actions associated with the service data:

  • An operator creates data and associates it with ME subsystems.
  • The SM subsystem maintains the primary copy locally and distributes data to ME subsystems.
  • Any further data updates are distributed automatically to the associated list of ME subsystems, whenever an update is done on the SM subsystem.

These are the actions associated with the subscriber or subscription data:

  • A user creates the data and associates with an ME subsystem, or SM subsystem employs algorithms to choose an optimal host.
  • SM subsystem provides the conduit for data to create, read, update and delete, but maintains locally only the ME subsystem location of the data.

These are the actions associated with the SM subsystem when it acts as an API gateway:

  • The SM subsystem provides methods for subscriber provisioning.
  • The SM subsystem determines the appropriate NCC subsystem and proxies the request.

For more information about the SM GUI, provisioning, and data distribution, see the respective NCC documentation.
