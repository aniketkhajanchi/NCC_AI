Pre-installation

NCC is deployed into your Kubernetes environment. Before the installation of NCC, these items are required:

  • The Kubernetes cluster is ready. For more information, see Cloud infrastructure prerequisites.
  • Istio service mesh is set up. NCC can install the service mesh if it is not already part of the platform.
  • Domain Name Server (DNS) entries for the system endpoints are available. These depend on the number of systems installed, and whether multiple networks for different types of traffic are used.
  • TLS and mTLS certificates are available for the system endpoint domains.
  • NCC container images are available in a container registry accessible to the Kubernetes cluster.
  • NCC Helm charts are available either in a Helm registry that is accessible to the Helm orchestrator or on disk. The Input Generator installation tool provides all of the Helm charts on disk.
  • Helm client version 3.2 or greater is available.
  • These items are available to run the Input Generator tool that is used to prepare the Helm chart inputs:
    • NCC input-generator container image
    • A container runtime environment such as Docker or Podman
    • Bash command line interface
    • Excel or LibreOffice version 6.2 or later for editing input spreadsheets
  • The host FQDN and User and Role Management FQDN are available for the OSP GUI, which is used to define the SS7 configuration and routing, if the SS7 subsystem is a part of the NCC deployment.
  • A secret for communication with ETCD is available, if the SS7 subsystem is a part of the NCC deployment.. For more information, see Secret for communication with ETCD.

Secret for communication with ETCD

SS7 Helm chart requires a secret with the name etcd-sdc-client-cert-secret. This secret must be available so that it can be used during deployment.

Note:
Make sure that the deployment is done on a clean setup. No stale data can be present from previous deployments. If the NCC SS7 and SS7 MariaDB Helm charts are redeployed on the same setup on which it was previously deployed, then check whether persistent volume claims exist from the previous deployment.

To check whether persistent volume claims from the previous deployment exist, use the following command:

$kubectl get pvc -n <namespace>| grep ss7

where <namespace> is the namespace in which helm charts are deployed.

For example, if you find the existing pvc related SS7 previous deployment, remove the existing pvc by entering:

for PVC in `kubectl get pvc -n <namespace> | grep ss7 | awk '{ print $1 }'`;do kubectl delete pvc ${PVC} -n <namespce>; done

where <namespace> is the namespace in which helm charts were deployed.

Cloud infrastructure prerequisites

The deployment environment for NCC is the Kubernetes cluster. The Kubenetes cluster is ready for NCC installation when the following is done and available.

  • Sufficient Kubernetes cluster resources are available.
  • Sufficient OpenStack cloud resources are available.
  • Security requirements are configured.

Dimensioning

Nokia provides dimensioning and engineering guidelines to configure cloud infrastructure resources such as RAM, vCPUs, and storage.

Role-based access control requirements

The application uses Kubernetes APIs controlled through RBAC security. NCC helm charts create their own service accounts that request only the permissions required for that function.
