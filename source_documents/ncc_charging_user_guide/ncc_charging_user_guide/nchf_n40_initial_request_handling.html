NCHF (N40) initial request handling

This feature provides duplicate session detection capabilities when processing N40 create transactions. This feature enables NCC to handle the following scenarios:

  • Duplicate create sessions, which may be sent within a relatively short interval from one or more SMFs when establishing a new PDU session
  • Cases where an SMF has locally stored the maximum number of containers allowed due to assume positive processing and, subsequently, the SMF sends another create with the same chargingId, dnnId, sNSSAI, and 3gpp-Sbi-Origination-Timestamp used for the initial session.

Configuration

  • Enable the feature by setting application preference Enable Duplicate Diameter Initial Request Handling to NCHF or NCHF_GY_AND_RO
  • Use N40 Create Duplicate Session Detection Interval (Seconds) to provide a value for the duplicate session detection interval

Important points

  • Identification of Update Transaction: If InvocationSequenceNumber of request is equal to InvocationSequenceNumber of stored session, then it implies that no update transaction is received for the session in past.
  • CDR is not generated for duplicate requests. Hence, timestamps and other information that changes with duplicate request handling is not recorded.
  • For duplicate requests, the rating or charging logic is re-executed. Hence, response or action may differ in some cases as per the rating condition.
  • Format of 3gpp-Sbi-Origination-Timestamp:
    • Encoding of the header follows the ABNF as defined in IETF RFC 7230 [31].
    • Format: "3gpp-Sbi-Origination-Timestamp" ":" day-name "," SP date1 SP time-of-day "." milliseconds SP GMTEXAMPLE: 3gpp-Sbi-Origination-Timestamp: Sun, 04 Aug 2019 08:49:37.845 GMT
  • If N40 create invocation time < Session start time, then the request is considered as if it lies within the specified N40 Create Duplicate Session Detection Interval (Seconds).
