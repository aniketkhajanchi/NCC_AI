Exhausting higher priority resources before the lower priority resources

If a bucket (higher priority resource) is reserved by one or more ongoing sessions, then new sessions are either rejected or the main balance (lower priority resource) is used before consuming from the bucket. In this case, the lower priority resource is used before exhausting the higher priority resource.

This feature provides the functionality to exhaust the higher priority resources before using the lower priority resources.
Note:
This feature is applicable only when main balance has the lowest priority.

For the reported usage quota, NCC re-evaluates the applicable resources, and commits the usage to the resources with higher priority even if they are reserved by other sessions. Lower priority resources are consumed only after higher priority resources are exhausted. This functionality is called First Come First Commit Resource.

If this feature is disabled, then lower priority resources can be exhausted before the higher priority resources.

Configuration

The functionality to exhaust the higher priority resources before using the lower priority resources can be enabled using either of the following:
  • Set the application preference First Come First Commit Resource to Enabled.

    Default: Disabled

  • Set First-Come-First-Commit-Resource attribute to Enabled in GY_PRE_PROCESSING trigger.

The First-Come-First-Commit-Resource attribute in GY_PRE_PROCESSING trigger has higher priority than the application preference First Come First Commit Resource. If the rule attribute is configured, then the application preference is ignored.

First Come First Commit Resource

NCC commits the reported USU from the applicable higher priority resource even if it is reserved by other sessions. Reported USU is committed until the higher priority resource is exhausted.

If the USU is completely committed to the higher priority resource, then the lower priority resource which was reserved by that session is released. If higher priority resource is exhausted, then the remaining usage quota is allocated from the lower priority resource.

Resource over-reservation

If a session reserves a lower priority resource, but commits to a higher priority resource later, then the higher priority resource is over-reserved.

Balance threshold for over-reservation

The balance threshold for over-reservation can be configured using the application preference Balance Threshold for Over Reservation and rule attribute Balance-Threshold-For-Over-Reservation in GY_PRE_PROCESSING trigger.

It allows main balance to be used for resource outstanding usage, if any.

The rule attribute Balance-Threshold-For-Over-Reservation has higher priority than the application preference Balance Threshold for Over Reservation. If the rule attribute is configured, then Balance Threshold for Over Reservation is ignored.

Note:
Configuring Balance Threshold for Over Reservation has the risk of generating revenue loss for main balance. When the session is reserving a bucket that is used by other sessions, and the rate for the outstanding usage for the session is higher than other sessions, then outstanding charge is discarded and stored in CDR.

Available balance calculation for quota allocation

If a resource is over-reserved while allocating quota, then only the delta balance (available main balance - Balance Threshold for Over Reservation) can be used for the allocation.

If secondary balance is applicable, then only the delta secondary balance (available secondary balance + available main balance - Balance Threshold for Over Reservation) can be used for the allocation.

Sending RAR while allocating quota

While allocating quota or committing usage, RAR is sent to the sessions which have reserved the over-reserved resources to force quota reporting, when the following conditions are met:

  • A resource is over-reserved.
  • Quota allocation fails because of insufficient credit.
  • Quota allocation/commit is successful but the available balance (including main balance and secondary balance, if applicable) is equal to or less than the value configured for Balance Threshold for Over Reservation.
    Note:
    If the over-reserved resource is at group level, then the rule attribute Group-Balance-Threshold-For-Over-Reservation in GY_PRE_PROCESSING trigger is checked for the available group balance. If the device belongs to a local group, then device balance is equal to or less than the Balance Threshold for Over Reservation.
  • The application preference Trigger RAR for Over Reservation or rule attribute Trigger-RAR-For-Over-Reservation in GY_PRE_PROCESSING trigger is set to any value other than None. The session which has reserved the over-reserved resources is either data call or IMS call.
    Note:
    The rule attribute Trigger-RAR-For-Over-Reservation has higher priority than the application preference Trigger RAR for Over Reservation. If the rule attribute is configured, then Trigger RAR for Over Reservation is ignored.
Consider the following important points about sending RAR while allocating quota or committing usage:

  • For a data session, RAR is sent only to the rating groups/categories that have reserved the over-reserved resource.

  • The application preference Interval for Trigger RAR to same RG is used to configure the interval within which a RAR is sent only once to an IMS session/rating group. It is used to avoid sending RAR multiple times to the same IMS session/rating group within the configured interval. If the interval is zero, then there is no limitation on sending RAR multiple times to an IMS session/rating group.

  • If RAR has been sent to an IMS session/rating group, and the IMS session/rating group is waiting for usage report (CCRU/CCRT), then RAR is not sent again until the usage report is received.

  • If a bucket in the bundle queue is over-reserved, the related bundle in the queue is not the last bundle, and the application preference RAR for Queue Bundle Over Reservation is set to Disabled, then RAR is not sent for the bucket.

  • For calls (both session calls and IEC calls), even if the over-reserved resource is not selected by the current call, and the available balance is lower than the Balance Threshold for Over Reservation, RAR is triggered for sessions/rating groups which use the over-reserved resource.

Additional information

  • The feature can cause revenue loss for main balance.
  • This feature is not applicable to event charging (for example, bundle renewal fee, step fee). Hence, event charging can be done even when the available balance is less than the Balance Threshold for Over Reservation. This may lead to insufficient balance for over-reserved resource outstanding usage.
  • Step up fee, BOU activation fee, and bundle renewal fee can be applied even if the balance is less than the value configured for Balance Threshold for Over Reservation. This may lead to insufficient balance for bucket outstanding usage.
  • If a queued bundle is activated during reservation, then it cannot be rolled back. Hence, with the First Come First Commit functionality, the next in-active instance in the queue can be activated before completely exhausting the current active instance of the queue.
