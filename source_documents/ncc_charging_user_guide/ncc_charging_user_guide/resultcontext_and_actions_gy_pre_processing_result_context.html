Tariff-Time-Change-Multiple GY_PRE_PROCESSING This result context can be used to configure the multiple tariff change time points.
Trigger-Life-Cycle-Event GY_PRE_PROCESSING The value of this attribute can be set to a custom string based on the type of traffic (service context) received.
Rate-RoundingFactor GY_PRE_PROCESSING This action provides a rounding mechanism to charge the calls with a configurable rate rounding factor at Global (RSV) or Tariff Rate level. Cost rounding is applicable only for monetary resources (that is, main balance and money buckets). This rounding mechanism is over and above the existing NCC rounding mechanisms for calculation precision and database precision (defined in the application preferences Calculation Precision and DB_PRECISION).
Note:
It is recommended to apply it only in one of the trigger points (Tariff, Gy_Pre_Processing, or Gy_Credit_Request).
Valid-Call-Correlation GY_PRE_PROCESSING This action marks leg1 of a voice mail call as eligible for correlation.
Establish-Correlation GY_PRE_PROCESSING This action establishes correlation between IMS sessions and triggers correlated charging.
Enable-Silent-TTC GY_PRE_PROCESSING This action enables the silent TTC.
Override-Overage-Limit GY_PRE_PROCESSING This attribute overrides the overage limit.
Reset-Gy-CDR-Aggregation GY_PRE_PROCESSING This action specifies a new TTC time, which is used by the CGF to stop and start the Gy aggregation logic. It is used when the customer would like to reset the Gy aggregation logic for example at every midnight.

NCC generates root level tag in CDR: resetAggregation = true
First-Come-First-Commit-Resource GY_PRE_PROCESSING

This parameter is used to enable the functionality to exhaust the higher priority resources before using the lower priority resources.

It can take the following values:
  • Enabled
  • Disabled

This parameter has higher priority than the application preference First Come First Commit Resource. If this parameter is configured, then First Come First Commit Resource is ignored.
Balance-Threshold-For-Over-Reservation GY_PRE_PROCESSING This parameter defines the threshold for account balance. If a bucket or counter is over-reserved, then the balance threshold defined in this parameter cannot be exceeded.

This parameter has higher priority than the application preference Balance Threshold for Over Reservation. If this parameter is configured, then Balance Threshold for Over Reservation is ignored.
Trigger-RAR-For-Over-Reservation GY_PRE_PROCESSING

This parameter is used to define whether RAR should be triggered for IMS or data calls when the applied bucket or counter is over-reserved and the balance threshold configured in the application preference Balance Threshold for Over Reservation is reached.

This parameter has higher priority than the application preference Trigger RAR for Over Reservation. If this parameter is configured, then Trigger RAR for Over Reservation is ignored.
Group-Balance-Threshold-For-Over-Reservation GY_PRE_PROCESSING

This parameter is used to define the threshold for group account balance. If a group bucket or counter is over-reserved, then the balance threshold defined in this parameter cannot be exceeded.

This parameter has higher priority than the application preference Balance Threshold for Over Reservation. If this parameter is configured, then Balance Threshold for Over Reservation is ignored.
Note:
A operator can also configure the Tariff-Time-Change, which is an optional parameter and the time is specified in hour:min:sec for the 24 hour format.

Overriding overage limit

The Override-Overage-Limit attribute in the GY_PRE_PROCESSING trigger is used to override the overage limit of the account.

The change is temporal (applicable only for a particular processed CCR), and it is not saved into the database.

For example:
Table: Overriding overage limit
If the Override-Overage-Limit is.. Then Result
0 The overage limit remains 0. The existing overage limit is overridden.
-5 The new overage limit is -5. The existing overage limit is overridden.
9999999999 The overage limit is overridden to unlimited overage limit. The existing overage limit is overridden.
null Overriding is not applicable. The existing overage limit is considered.
