Defining splitting criteria

Splitting criteria are reusable expressions which can be used by any number of file groups. Criteria are used for matching output records to correct distribution rules.

To create or modify criteria, update the ‘Split Criteria’ lookup table.

To update the dataset configuration used by split criteria (column ‘Dataset’), update the dataset lookup table, either through the UI or by importing data as described in Updating lookup tables using update_db_table.sh.

Table: Columns in ‘Split Criteria’ lookup table
Column name Description Valid value Optional
Criteria Id

All criteria must have a unique identifier so they can be used by file groups.

 Any unique string. No
Output Field Name

The name of the field whose value is used for the comparison. Any field level can be used.

Field name present in the output record.

If the field is not at the root level, full field hierarchy should be used.

Example: listOfMultipleUnitUsage.MultipleUnitUsage.resultCode

Note:
If the field is situated under a list with multiple instances, its first occurrence will be considered for filtering.

If the field is not present, the criteria is evaluated as false.

 No
Criteria Type

The type of comparison.

 See Criteria types and values. No
Criteria Value

For simple comparisons, this is the fixed value that the record field value is compared against.

 See Criteria types and values.

Empty if this is a dataset comparison. Otherwise mandatory.
Dataset

For dataset comparisons, this is the name of the lookup table that contains the dataset to be used.

 The name of an existing lookup table that contains a single column.

Empty if this is a simple comparison. Otherwise mandatory.
Note:

Because Charging Gateway Function does not support empty values in lookup tables, they must be specified by using the dash (-) character.

Table: Criteria types and values
Criteria type Description Valid values
EXACT_MATCH A strict case-sensitive comparison.

Any string.

Example: abc
REG_EX_MATCH

A pattern match against a regular expression.

For more information on the allowed regular expressions, see the documentation for java.util.regex.Pattern.

A valid regular expression.

Example: abc.*def

REG_EX_NO_MATCH

The opposite of REG_EX_MATCH.

A valid regular expression.

Example: abc.*def
GREATER_THAN A numeric comparison.

A valid integer value.

Example: 1
LESS_THAN A numeric comparison.

A valid integer value.

Example: 1000
DATA_SET_EXACT_MATCH

Check that the field value matches (case-sensitive) any entry in the specified lookup table.

The name of an existing lookup table that contains a single column.

Example: LT_MSISDN_LIST

DATA_SET_NO_MATCH

Check that the field value (case-sensitive) does not match any entry in the specified lookup table.

The name of an existing lookup table that contains a single column.

Example: LT_MSISDN_LIST
DATA_SET_STARTS_WITH_MATCH

Check that the field value starts with (case-sensitive) any entry in the specified lookup table.

The name of an existing lookup table that contains a single column.

Example: LT_MSISDN_LIST
