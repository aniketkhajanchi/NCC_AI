VT randomization when TTC is disabled for 4G and 5G calls

For the 4G and 5G calls, NCC provides the provision to randomize Validity Time (VT) values at BCR if Tariff Time Change (TTC) is disabled for the bundle.

If this feature is turned on, and the standard Validity Time (VT) value spans the renewal date for a subscription to a bundle when TTC is disabled, then NCC randomizes the VT for CCR-I, CCR-U, Nchf_ConvergedCharging_create, and Nchf_ConvergedCharging_update transactions.

Configuration

To enable VT randomization when TTC is disabled, the following configuration takes effect:

Application preferences

  • VTAF Postpaid Without TTC: This application preference is used to randomize VT value present in the returned CCA message for GY session. It defines the VT adjustment randomization factor for postpaid account when TTC is disabled.

    Note:
    If the value of this parameter is zero, then VT computation / randomization is done as per the applicable formulas.

  • VT Adjustment Factor Prepaid: This application preference defines the VT adjustment randomization factor for prepaid account when TTC is disabled.

  • AutoGen SPR Ids: This parameter is used to enable the automatic generation of IDs for subscriber model entities (Device, Account, User and Group).

  • EDR Generator Flag: If this parameter is set as true, then EDRs are generated.

    Default: false

  • EDR Logging Enabled: If EDR logging is enabled, then value becomes case insensitive.

    Default: false

Rule configuration

GY_PRE_PROCESSING.Is-TTC-Enabled should be configured. It is available in GY_PRE_PROCESSING trigger.

Is-TTC-Enabled attribute should be set to true.

Service (Bundle) configuration

  • Disable Tariff Time Change parameter in the bundle is enabled.

  • Randomize validity time without tariff time change parameter in the bundle is enabled.
    Note:
    This parameter is available only when Disable Tariff Time Change parameter is enabled.

  • bcrReportingTime is not set for the device.

VT calculation

If the subscription is funded by a prepaid account, then the validity time (VT) is calculated as follows:
  • VT = time to the BCR (“T1”) + RAND [1 second, MIN [VTAFprepaid, T2-T1], where T2 is the time of the next subscription activation/renewal/expiration or TOD event.
If the subscription is funded by a postpaid account, then the VT is calculated as follows:
  • VT = time to the BCR (“T1”) + RAND [1 second, MIN [VTAFpostpaid_without_TTC (as per REQ-0010), T2-T1], where T2 is the time of the next subscription activation/renewal/expiration or TOD event

If the application preference Offline Session Control is enabled, then the time at which the VT expires is stored as bcrReportingTime of the device.

For Gy CCA-I or CCA-U responses, the value calculated using the VT randomization formula based on prepaid or postpaid account is returned to the network as the Validity Time (VT).

For N40 create or update responses, if Calculation of BCR reporting time and offline TTC feature is enabled, then the VT value returned to network is the addition of VTBCRoffset application preference value and the value calculated using the VT randomization formula based on prepaid or postpaid account.
Note:

For multiple subscriptions whose event time is T1/T2 which is same as following event time - one subscription renewal time is at T1 and second subscription is at T2.

The VT randomization logic as per this feature occurs if at least one of the subscription, the disabled TTC and VT randomization flag is TRUE/TRUE, and none of the TTC and VT randomization flag is TRUE/FALSE.

Calculation of BCR reporting time and offline TTC feature should be enabled with the following configuration:
  • Offline Session Control application preference is set to Enabled for both account BCR and subscription BCR.
  • VTBCRoffset application preference is set to 5 seconds.
For details on Calculation of BCR reporting time and offline TTC, see Calculation of BCR reporting time and offline TTC.

Use cases

Prerequisites:

Consider the following configuration for all the use cases:

  • Min Spread is set to 20 seconds.

  • VT Adjustment Factor Prepaid is set to 1800 seconds.

  • VTAFpostpaid_without_TTC is set to 3000 seconds.

  • VT Adjustment Factor is set to 1500 seconds.

  • TTC Adjustment Factor is set to 900 seconds.

  • TTC Adjustment Factor Large is set to 3000 seconds.

  • Offline Session Time Period is set to 1000 seconds.

  • VTBCROffset is set to 5 seconds.

  • Validity Time (VT) in slicing profile is set to 43200 seconds.

  • GY_PRE_PROCESSING.Is-TTC-Enabled in GY_PRE_PROCESSING trigger is set to true.

Use case 1: Device with prepaid account bundle

Objective: Check that Validity Time (VT) is randomized when Disable Tariff Time Change and Randomize validity time without tariff time change flags are set to true for a prepaid subscription.

Configuration:

  • Offline Session Control is set to AccountBCR_SubscriptionEvent.

  • In Bundle1, Disable Tariff Time Change and Randomize validity time without tariff time change flags are set to true. Bundle has periodic lifecycle of 60 minutes.

  • Device subscribes to Bundle1 using prepaid account.

Execution:

Execute the Gy-Initial call. Validity Time (VT) should be randomized using the following formula:

VT = time to the BCR (“T1”) + RAND[1 second, MIN[VTAFprepaid, T2-T1]

Verification:
  • Verify that TTC is not sent.
  • Verify that BCR reporting time is set.
  • Verify that VT is set as time to BCR reporting time.
Use case 2: Device with postpaid account bundle

Objective: Check that Validity Time (VT) is randomized when Disable Tariff Time Change and Randomize validity time without tariff time change flags are set to true for a postpaid subscription.

Configuration:

  • Offline Session Control is set to AccountBCR_SubscriptionEvent.

  • In Bundle1, Disable Tariff Time Change and Randomize validity time without tariff time change flags are set to true. Bundle has periodic lifecycle of 60 minutes and a bucket of 1000 Bytes.

  • Device subscribes to Bundle1 using postpaid account.

Execution:

Execute the Gy-Initial call. Validity Time (VT) is randomized using the following formula:

VT = time to the BCR (“T1”) + RAND[1 second, MIN[VTAFpostpaid_without_TTC, T2-T1]

Verification:
  • Verify that TTC is not sent.
  • Verify that BCR reporting time is set.
  • Verify that VT value is set as time for BCR reporting time.
Use case 3: Device with multiple postpaid account bundles and no PC status change at BCR
Objective: Check that Validity Time (VT) and Tariff Time Change (TTC) are randomized when postpaid account has two bundles with the following configuration:
  • Bundle1 with Disable Tariff Time Change and Randomize validity time without tariff time change flags set to false.
  • Bundle2 with Disable Tariff Time Change and Randomize validity time without tariff time change flags set to true.
Configuration:

  • Offline Session Control is set to Disabled.

  • GY_PRE_PROCESSING.Is-TTC-Enabled in GY_PRE_PROCESSING trigger is set to true.

  • Bundle1 has periodic lifecycle of 60 minutes and a bucket of 50 Bytes.
  • Bundle2 has periodic lifecycle of 60 minutes and a bucket of 1000 Bytes.

  • Device subscribes to Bundle1 and Bundle2 using postpaid account.

Execution:

Execute the Gy-Initial call with RSU =100.

Verification:
  • Verify that TTC is randomized according to Disable TTC for prepaid accounts and alignment of TTC and VT for postpaid accounts.
    Note:
    For details, see Disable TTC for prepaid accounts and alignment of TTC and VT for postpaid accounts.
  • Verify that Validity Time (VT) is randomized using the following formula:

    VT = time to the BCR (“T1”) + RAND[1 second, MIN[VTAFprepaid, T2-T1]

Use case 4: Device with multiple postpaid account bundles
Objective: Check that Validity Time (VT) is not randomized when postpaid account has two bundles with the following configuration:
  • Bundle1 with Disable Tariff Time Change set to true and Randomize validity time without tariff time change set to false.
  • Bundle2 with Disable Tariff Time Change and Randomize validity time without tariff time change flags set to true.
  • T1 and T2 have different timestamps.
Configuration:

  • Offline Session Control is set to Disabled.

  • Bundle1 has periodic lifecycle of 60 minutes and a bucket of 50 Bytes.
  • Bundle2 has periodic lifecycle of 60 minutes.

  • Device subscribes to Bundle1 and Bundle2 using postpaid account.

Execution:

Execute the Nchf create request with RSU =100.

Verification:
  • Verify that TTC is not sent.
  • Verify that Validity Time (VT) is not randomized and is aligned to T1.

Additional information

Hard-stop feature with GSU=0 and FUI=restrict-access returns default VT as per existing NCC default behavior, which is Call Time (Update ) < BCR(T1) <TOD Start Time(T2) < TOD End Time.

Update comes before BCR (T1) Hard-Stop is reached. VT is sent as default Gy Session Inactivity Duration (seconds) =86400. (This behavior is changed - VT Randomization occurs as per bundle configuration with T1 as BCR and T2 as TOD start time.)

BCR(T1) < Call Time (Update ) < TOD Start Time(T2) < TOD End Time Or

Call Time (Update) = BCR < TOD Start Time(T1) < TOD End Time (T2).

If the update happens at BCR (T1) or before TOD Start Time (T2), VT is sent as default Gy Session Inactivity Duration (seconds) = 86400 (Since Hard-Stop is reached, VT is sent as time to start TOD, GSU=0, FUI=restrict-access.

  • This feature follows the updated formulas if Bundle VT randomization flag is TRUE even if subscription has reached its max renewal limit.
  • If this feature is not turned on, then the existing behavior (that is, VT randomization for new data sessions) for VT randomization is applicable. For details, see VT randomization for new data sessions.
  • Randomization of VT values when TTC is disabled is applicable only for subscriptions.
  • This feature is not applicable to the suspended subscriptions.
  • UC TOD events are not included in BCR TEM randomization process at provisioning.
  • If a subscription is deleted, then re-randomization of existing impacted BCR TEM timer is not considered.
  • The periodicity of account or subscription should not be less than the configured value of VTAFprepaid application preference.
Note:
  • The feature is not applicable for device barred and subscription scenarios. For subscription barred and device barred NCC exit sign logic is executed - IMS/Gy Session Inactivity Duration (seconds).
  • It is recommended to configure RAR in all provisioning calls, so that any change in BCR timer randomization can be handled by the network.
  • For feature applicability and recommendation it is expected that the operator enables the Disable TTC flag and enable the VT randomization flag as TRUE/FALSE respectively.
