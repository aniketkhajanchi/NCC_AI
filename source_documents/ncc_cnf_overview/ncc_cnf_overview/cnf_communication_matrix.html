Secure TLS communication modes

Previously, the NCC implementation of mTLS used ISTIO_MUTUAL. This assumes that to communicate to another Istio-enabled application that shares the same root certificate you must use Istio. You can now use your own generated certificates/secrets for TLS communication with TLS mode set to MUTUAL which does not need the external services to be Istio-enabled or require the same CA.

NCC supports SIMPLE, ISTIO_MUTUAL, and MUTUAL TLS modes in the Input Generator spreadsheet.

Note these assumptions:

  • You can either choose ISTIO_MUTUAL or MUTUAL for mTLS communication, not both.
  • You must create the required certificates, keys, and setup the Kubernetes secret for both the client and server whose reference is passed in the Input Generator under the Istio and Shared sections for all namespaces.
  • The mTLS FQDN value in the Input Generator is considered for both ISTIO_MUTUAL or MUTUAL mode.
  • Only ingress and egress (including inter-system traffic) uses your CA certificates and keys. All internal workload certificates are generated and assigned by Istio. These certificates are self-signed.
  • All GUI (SM, Prometheus, Metrics Viewer, CGF [DR]) traffic accessed by a web browser is TLS and not mTLS.
  • Different client certificates are set for different types of traffic in the Shared section for all namespaces.
  • All secrets containing server certificates for ingress traffic are set in the Istio section for all namespaces.
  • All TLS traffic is handled via istio-proxy.
  • ISU is supported from MUTUAL TLS to MUTUAL TLS. ISU and NSU are not supported for SIMPLE to MUTUAL TLS.
  • The default option for TLS mode is SIMPLE. If you need to use MUTUAL mode, you must select that value in the Input Generator spreadsheet.
  • Communication of external Kafka with Fluentd uses TLS mode SIMPLE only.
  • Service Entries with client secrets for egress communication are set in Service Entries section for all namespaces
