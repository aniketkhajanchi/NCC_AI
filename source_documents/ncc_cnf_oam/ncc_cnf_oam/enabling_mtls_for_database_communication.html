Enabling TLS for intersystem Aerospike XDR communication

This procedure provides instructions on how to enable TLS protocol for Aerospike XDR communication between the two geo-redundant systems of the same type. This procedure is used post-installation when NCC has already been installed. For instructions on how to enable TLS for intersystem Aerospike communication during installation, see the Installation and Upgrade Guide. In this procedure, the two systems are called System 1 and System 2.

Note:
Before you begin this procedure, ensure that the xdrproxy is enabled.

  1. Manually expose a secure port, for example 3009, on Ingress gateway.
  2. In the input file on System 1 and System 2, set the xdrproxy.mtls.mode and aerospike.network.outboundXdr.mtls.mode parameters to SIMPLE, and the xdrproxy.mtls.credentialName parameter to any existing credential name, for example, sps-credential.
  3. On System 1set the xdrproxy.mtls.enabled parameter to true in the input file and then run helm upgrade with -f option passing the input values.yaml file.

    Expected output

    This allows the xdrproxy pods to receive both TLS and non-TLS traffic from Aerospike on System 2.

  4. On System 2 set the xdrproxy.mtls.enabled parameter to true and the aerospike.network.outboundXdr.mtls.enabled to true in the input file and then run helm upgrade with -f option passing the input values.yaml file.

    Expected output

    Aerospike pods restart, and during the restart both TLS and non-TLS traffic goes to System 1 xdrproxy. After the restart, there is only TLS traffic.

  5. On System 1 set the aerospike.network.outboundXdr.mtls.enabled parameter to true in the input file and then run helm upgrade with -f option passing the input values.yaml file.
  6. When the related pods are fully restarted on both sites, to remove the insecure port from xdrproxy, on System 1 and System 2 set the xdrproxy.exposeInsecure parameter to false in the input file and then run helm upgrade with -f option passing the input values.yaml file.
