Combining UC limit of multiple counters

You can combine usage for multiple Counters (for example, data usage for 4G and 5G counters). That is, a single combined usage control limit applies on usage for multiple reference counters. Use the parameters Counter Name and Referenced counter IDs.

After the combined usage limit is applied, the attributes Counter Name and Referenced counter IDs become visible in Device overview. However, Referenced counter IDs only shows the counter(s) attached to the subscribed bundles, and not all defined in UC definition.

The application preference Enable UC Sync For API is available for enabling UC synchronization with the UC definition through the read APIs for Device, Account, Subscription, and Profile. Refer to the Global Configuration Guide for more details about this application preference.

Lifecycle and reset for multi-reference self-counter

Multi-reference self-counter accumulates usage of other counters, and it does not have its own lifecycle. It does not reset with lifecycle or for the bundle from which the consumption is being done. Reset is governed by the reference counters, that is, when any of the reference counters for the multi-reference self-counter is reset, the value is modified according to it. On Counter renewal or reset when the current value is moved to the previous value, the previous value is also updated by accumulating previous values of the referenced counters.

Assumptions and limitations

  • For slice reduction, the updated value of UC multi-reference self-counter after committing is used. In case of Notification stop on multi-reference self-counter it works as Notification continue.
  • Even if applicability is configured in UC, the multi-reference self-counter sums up all counters mentioned in reference irrespective of the different applicability conditions of reference counters.
  • Policy counter and SNR are not supported.
  • Reserved quota is NOT considered when allocating new quota based on any counter’s current value (including the multi-reference self-counter).
  • Multi-reference self-counter value is considered for slice reduction even in the case of No-Charge and all other resource types.
  • Do not applicability condition of the multi-reference self-counter to accumulate counter applicable to the same RGs as it will lead to a double addition in the consumption.
  • If the definition of UC is modified, RAR is not generated for any session.
  • If any usage service is configured with self-counter and old reference counter, then its behavior is not affected by combined UC limit.
  • As multi-reference self-counters have references of multiple counters (for example, reference counters shared between parents and child), and there may be scenarios where the reference counter continues to increase even when the threshold has reached for the UC self-counter, as a result, the UC self-counter value can be more than the limit.
  • Final subscription counters are not considered in the multi-reference counter.
  • Barred subscription counters are considered in the multi-reference counter.
  • In patch API to update UcServiceInstance, list of reference counters cannot be modified.
  • The following are not supported for UC multi-reference self-counters:
    • Overage
    • Group counters
  • Definitions cannot be updated by replacing the old reference counters with the old self-counter.
  • Only reference counters can be selected for accumulation in multi-reference self-counter.
  • During UC definition if TPG and limit are updated, then it is not considered. If UC is already subscribed the instance must be updated to update TPG and limit.
  • Do not configure announcement for SMS/MMS calls in GY_PRE_PROCESSING and UC_PROCESSING triggers. Announcement is supported for VSA AVPs.
