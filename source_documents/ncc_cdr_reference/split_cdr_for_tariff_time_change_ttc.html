Split CDR for tariff time change (TTC)

The split CDR for TTC events is applicable in the following scenarios:
  • CDR Generated per MSCC is true
  • CDR Generated per MSCC is false, and Split TTC CDR is ENABLED.

The following tags are added to the CDR or have different value for TTC.

Tag

Description

  
changeCDRReason

Only applicable when tariff-time change (TTC) is enabled at MSCC level.

In case of TTC, two CDRs are generated, that is, one for before and one for after. The changeCDRReason tag is added for split CDR and it is available only for “Before CDR” with value “BCR”.

  
tariffTimeChange This tag is generated for MSCC CDRs during commit of the resources. The TTC sent during the previous CCA is added to this tag for before TTC CDRs.  
recordSequenceNumber In case of CDR per MSCC, recordSequenceNumber for the after CDR is increased by 1 from the recordSequenceNumber of the “before CDR”.  
recordOpeningTime

In case of CDR per MSCC for “before usage CDR”, recordOpeningTime contains the time (taken from either the IMS-Information.Time-Stamps.SIP-Request-Timestamp AVP if present, or Event-Timestamp if present, else platform time is used) of that CCR which was received by NCC for a session whose CDR was generated and sent in previous CCA.

In case of CDR per MSCC for “after usage CDR”, the recordOpeningTime is equal to the TTC time.

 29/06/2018 22:13:20
recordClosureTime

In case of CDR per MSCC for “before usage CDR”, recordClosureTime is equal to the TTC time.

In case of CDR per MSCC for “after usage CDR”, recordClosureTime is the time taken from either the IMS-Information.Time-Stamps.SIP-Request-Timestamp AVP if present, or Event-Timestamp if present, else platform time is used.

 29/06/2018 23:13:20
totalVolConsumed The total consumption is split into both before and after CDRs.  
uplinkVolConsumed The uplink consumption is split into both before and after CDRs.  
downlinkVolConsumed The downlink consumption is split into both before and after CDRs.  
BucketInfo The values for tags (bucketBalanceBefore, bucketBalanceAfter, bucketCommitedUnits, bucketReservedUnits) are split into both before and after CDRs.  
duration

Call or record duration of the split records as per the recordOpeningTime and recordClosureTime.​

  

Important points to consider

  • This feature extends the CHF TTC CDR Sequence Numbers logic to assure that CDR sequence numbers continue to be sequentially arranged and/or when real-time or APR transactions are received for a session which was terminated abnormally.
  • This includes support for sequentially numbering multiple CDRs generated if an APR transaction includes TTC parameters with before and after usage .
  • In case of a TTC split, enable the parameter Split TTC CDR.

    If an active Ro or Gy session receives an APR CCR-T transaction the CHF must use the stored next CDR recordSequenceNumber for the session. If the transaction contains TTC trigger (TTC information), it results in creation of two CDRs before usage and after usage.

  • If the control flag for CHF TTC CDR Sequence Numbers is enabled then whenever an active Ro, Gy, or N40 session moves from active to abnormally terminated or enhanced N40 abnormally terminated the CHF must save the next sequential recordSequenceNumber to be used for a CDR generated session. For cases involving integration with the feature enhanced N40 abnormally terminated, the parameter N40 APR out of order Request Handling should be set either to ENABLED_WITH_CREATE_ISN_0 or ENABLED_WITH_CREATE_ISN_1 depending on use cases.
  • If the control flag for feature CHF TTC CDR Sequence Numbers is enabled then whenever the CHF receives an uncorrelated Ro, Gy, or N40 transaction it must use a recordSequenceNumber of n+1 in any CDR it generates due to receiving the transaction. If the transaction included TTC information resulting in two CDRs being generated then the second CDR generated (for any AFTER usage) must have a recordSequenceNumber of n+2. (n is the previous RSN)
  • In case of an APR TTC Split CDR, the recordEventResult CDR tag value does not dump as 15 (session spanned across BCR).
